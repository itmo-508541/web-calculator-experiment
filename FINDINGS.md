# FINDINGS — выводы эксперимента для развития GitLab AI Helper

Документ предназначен для **ИИ-агента**, который дорабатывает проект **GitLab AI Helper** — backend-сервис для верификации Issue и Merge Request в GitLab с помощью ИИ, включая MCP-сервер для чтения и изменения требований (Issue) из облачной LLM.

Здесь зафиксированы проблемы, обнаруженные в ходе эксперимента по No-/Low-coding разработке ПО: путь от пожеланий заказчика через роли «аккаунт-менеджер» и «бизнес-аналитик» до спецификации требований и переноса их в GitLab Issue. **Конкретные решения в документе не предлагаются** — только описание разрывов и ожидаемый **вид** того, чего не хватает в системе.

Документ **самодостаточен**: для понимания контекста не требуются другие файлы.

---

## Контекст эксперимента (кратко)

Эксперимент воспроизводит цепочку ролей: сбор информации от заказчика → бизнес-анализ и спецификация требований → (далее) постановка задач разработчику → разработка.

На этапе бизнес-анализа получена спецификация из 29 пронумерованных требований (`REQ-001` … `REQ-029`) к сетевому калькулятору, включая:

- программный интерфейс (API) и веб-страницу режима Basic;
- эталон поведения Ubuntu 24.04 Basic;
- визуальные эталоны (скриншоты: основной вид, hover, press, ошибки);
- сценарии приёмки и критерии готовности v1.

Пользователь переносит требования в GitLab Issue **вручную** или через MCP-сервер GitLab AI Helper. В ходе переноса выяснилось: система **не умеет работать с вложениями (uploads)** в Issue. Визуальные эталоны пришлось выложить во внешний репозиторий; в тексты требований вставлены прямые URL на raw-файлы, чтобы отдельное требование оставалось **полным** даже при изъятии из общего списка.

В описании GitLab AI Helper свойство **модифицируемости** требований (IEEE 830: связная структура, оглавление, перекрёстные ссылки, раздельное выражение каждого требования) считалось **автоматически достижимым** как побочный эффект проверок однозначности, полноты, непротиворечивости и проверяемости отдельной Issue. Эксперимент показал, что это предположение **неверно**.

## F-001. Модифицируемость ошибочно принята за побочный эффект

### Проблема

Модифицируемость **совокупности** требований не возникает сама по себе от качества текста отдельных Issue. При переносе единой спецификации в **однородный список Issue** без типизации теряются оглавление, группы, иерархия и единая картина зависимостей.

### Проявление в эксперименте

- Спецификация имела оглавление, девять групп требований (G1–G9), перекрёстные ссылки `REQ-…`, приложения со сценариями проверки и критериями приёмки релиза.
- После разнесения по отдельным Issue структура **исчезла**: каждая Issue видна изолированно.
- Правка одного требования (например, текста ошибки для квадратного корня из отрицательного числа) не сопровождается автоматическим указанием на **зависимые** Issue; связи `#N` в тексте есть, но **оглавления, указателя и обзора корпуса** нет.

### На какие свойства влияет

- **Модифицируемость** — основной удар.
- **Непротиворечивость** (глобальная) — при правке одного требования сложнее увидеть каскад по всей совокупности.
- **Полнота совокупности** — риск дублирования и расхождения формулировок при ручном разнесении.

### Желательный вид решения (без проектирования)

Система должна **явно** поддерживать модифицируемость совокупности требований, а не полагаться только на качество отдельной Issue. Ожидаемые способности:

- **обзор** корпуса требований (список, фильтры, группировка — не только знание номера Issue заранее);
- сохранение или восстановление **структуры** (уровни, группы, родитель–потомок) при хранении в GitLab;
- при изменении Issue — **сигнал** о затронутых связанных требованиях (по `#N` и, по возможности, по смыслу);
- различение ролей Issue (требование vs тикет), чтобы легитимные различия уровней не считались противоречием.

## F-002. Вложения Issue не участвуют в жизненном цикле требования

### Проблема

GitLab AI Helper оперирует **только текстом** Issue (заголовок, описание) и историей версий текста. **Вложения** (файлы, загруженные в Issue в GitLab) не учитываются при верификации Issue, не доступны через MCP и не попадают в локальный снимок требования.

Для требований, где эталон приёмки — **изображение** (макет, скриншот), это создаёт **инфраструктурный разрыв**.

### Проявление в эксперименте

- Бизнес-аналитик оформил визуальные требования (совпадение цветов и размеров, реакция на hover и press, оформление ошибки деления на ноль) со ссылками на эталонные PNG.
- Перенести изображения **внутрь** GitLab Issue через MCP **невозможно** — функционал uploads отсутствует.
- Обходной путь: внешний хостинг файлов + raw-URL в markdown-тексте Issue. Это компенсация ограничения системы, а не её свойство.

### На какие свойства влияет

- **Полнота отдельной Issue** — сценарий верификации Issue формально проверяет критерии приёмки в тексте, но **не может** убедиться, что визуальный эталон доступен вместе с требованием.
- **Проверяемость** — критерий «сравнить с эталонным скриншотом» без вложения или устойчивой внутренней ссылки **неисполним** из одного артефакта GitLab.
- **Модифицируемость** — при смене эталона нужно править URL во многих Issue; версионирование изображений не связано с историей Issue.
- **Отслеживаемость** — связь «требование → эталон-изображение» вынесена за пределы системы.

### Желательный вид решения (без проектирования)

Требование в системе должно представлять **текст + связанные артефакты** как единое целое. Ожидается:

- **приём событий** GitLab о добавлении, замене и удалении вложений Issue и сохранение метаданных (и при необходимости содержимого) в том же контуре, что и текст;
- **MCP-инструменты**: перечислить вложения требования, прочитать или скачать файл (с учётом типа и размера), при необходимости — загрузить или заменить вложение в Issue;
- **верификация Issue**: учитывать вложения при оценке полноты и проверяемости (визуальное требование без прикреплённого или явно связанного эталона — замечание к Issue).

## F-003. Самодостаточность Issue при изъятии из спецификации

### Проблема

При переносе требований из спецификации в отдельные Issue предполагается, что **каждая Issue самодостаточна**: из её текста понятны поведение, критерии проверки и контекст **без** чтения всего документа-спецификации. На практике самодостаточность **неравномерна**.

### Проявление в эксперименте

| Категория требований | Самодостаточность при переносе в одну Issue |
|----------------------|---------------------------------------------|
| API, изоляция пользователей, состояние | Высокая — текст достаточен |
| Визуальные (внешний вид, hover, press, ошибки) | Только при явных URL на внешние файлы |
| Поведенческие «как Ubuntu 24.04 Basic» | Низкая — нужен живой эталон или полный перечень сценариев внутри Issue |

Сценарий верификации Issue проверяет полноту **текста**, но **не требует**, чтобы внешние эталоны были **доступны из Issue** (вложение, артефакт в контуре GitLab, устойчивый идентификатор).

### На какие свойства влияет

- **Полнота** (отдельной Issue) — формально может быть удовлетворена при пустой отсылке «как в эталоне».
- **Проверяемость** — исполнитель и приёмщик не могут проверить требование, имея только Issue.
- **Корректность** — риск расхождения между формулировкой в Issue и эталоном вне системы.

### Желательный вид решения (без проектирования)

Верификация и MCP должны **различать** отсылки, которые делают требование проверяемым, и отсылки-заглушки. Желательно:

- правило или проверка: если критерий приёмки ссылается на файл, изображение или внешний эталон — артефакт **присутствует** как вложение Issue или как однозначно разрешаемая ссылка в контуре проекта;
- подсказка при создании или редактировании требования: что включить в текст или вложения для самодостаточности;
- для поведенческих эталонов — явное требование перечислить **контрольные сценарии** в теле Issue или связать Issue со сценарием через `#N`, а не отсылать к «сравнению с продуктом X» без операционализации.

## F-004. Разрыв между монолитной спецификацией и однородным списком Issue

### Проблема

Спецификация требований по IEEE 830 — **единый документ** с введением, границами версии, допущениями, приложениями. GitLab AI Helper опирается на модель **одной Issue = один текстовый объект** без типизации. Перенос из документа в список Issue **не сохраняет** метаданные уровня документа.

### Проявление в эксперименте

- Решения анализа (стиль API, механизм идентификации клиента, модель операций) **намеренно не** входят в пронумерованные требования, но нужны продукт-менеджеру и разработчику.
- Критерии приёмки релиза (пять блоков AC) и приложение со сценариями — **вне** отдельных Issue; при разнесении только REQ теряется контекст приёмки v1.
- Пользователь вынужден дублировать структуру вручную или держать «мастер-документ» вне GitLab.

### На какие свойства влияет

- **Модифицируемость** — нет единой точки входа в совокупность требований.
- **Полнота совокупности** — система не помогает сверить «все ли требования из спецификации заведены».
- **Отслеживаемость** — цепочка «спецификация → Issue» не зафиксирована в системе.

### Желательный вид решения (без проектирования)

Возможность **не сводить всё к плоскому списку** без потери структуры:

- типизация или метаданные Issue (требование / тикет / критерий приёмки / решение анализа / приложение);
- Issue-«контейнер» или эквивалент для границ версии и общих критериев приёмки со ссылками на дочерние `#N`;
- инструмент сверки: ожидаемый набор идентификаторов требований vs фактически заведённые Issue.

## F-005. Внешние URL как обходной путь ломают доверие к хранилищу

### Проблема

Вставка raw-URL на файлы во внешнем репозитории в текст Issue **имитирует** полноту требования, но создаёт зависимости: доступность внешнего хоста, неизменность ветки и пути, права доступа, отсутствие версионирования вместе с историей Issue.

### Проявление в эксперименте

- Для визуальной приёмки в Issue вставлены ссылки на эталонные PNG (основной вид калькулятора, hover, press, ошибка деления на ноль).
- Требование **формально полное** в тексте, но **фактически** эталон не принадлежит GitLab-проекту требований.
- Смена эталона или удаление внешнего репозитория **ломает** проверяемость без изменения текста Issue.

### На какие свойства влияет

- **Проверяемость**, **полнота**, **отслеживаемость**, **модифицируемость** — все через внешнюю зависимость.

### Желательный вид решения (без проектирования)

Система должна предпочитать **артефакты в контуре GitLab** (вложения Issue, файлы в репозитории проекта с явной связью в тексте). При обнаружении критичных внешних URL в критериях приёмки — **предупреждение** при верификации Issue. Критерий готовности: визуальное требование полно, если эталон **доступен из того же проекта**, что и Issue.

## F-006. MCP не закрывает разрыв переноса спецификации

### Проблема

MCP-сервер умеет читать, создавать и изменять **текст** Issue и смотреть историю. Перенос 29 требований из подготовленных текстов для системы хранения остаётся **ручным** или полуручным: нет пакетного создания, нет привязки вложений, нет восстановления структуры групп и связей «критерий приёмки ↔ сценарий ↔ REQ».

### Проявление в эксперименте

- Пользователь копирует тексты в Issue по одному.
- Связи `#N` между требованиями выставляются вручную после создания всех Issue.
- Перенос **трудоёмок** и источник ошибок (пропуск требования, неверная ссылка).

### На какие свойства влияет

- **Модифицируемость** — ошибки при ручном копировании.
- **Непротиворечивость** — неверные или отсутствующие `#N`.
- **Полнота совокупности** — пропуск отдельных REQ.

### Желательный вид решения (без проектирования)

Возможности, снижающие ручной труд при **импорте** структурированной спецификации в GitLab (точный дизайн не задаётся):

- создание нескольких Issue из структурированного ввода с сохранением идентификаторов (`REQ-001` …) в заголовке или метаданных;
- отложенная простановка связей `#N` после создания корпуса;
- загрузка вложений вместе с текстом требования;
- отчёт о расхождениях между ожидаемым набором и созданными Issue.

## F-007. Верификация Issue не учитывает зависимость от живого внешнего эталона поведения

### Проблема

Часть требований формулируется как «поведение совпадает с Ubuntu 24.04 Basic»: проверка предполагает **внешнюю программу** и пошаговое сравнение. Сценарий верификации Issue оценивает формулировку в тексте, но **не фиксирует**, что для приёмки нужен внешний эталон и перечень контрольных шагов **внутри** требования или связанных Issue.

### Проявление в эксперименте

- Требование о совпадении промежуточных состояний с эталоном: конкретные числа для многих операций не перечислены в Issue — они снимаются с Ubuntu при прогоне (допущение аналитика).
- Issue с текстом «как в Ubuntu» может пройти проверку полноты, оставаясь **непригодной** для разработчика без доступа к эталону и сценариям.

### На какие свойства влияет

- **Проверяемость**, **однозначность** (на практике), **полнота** отдельного требования.

### Желательный вид решения (без проектирования)

При обнаружении отсылки к **именованному внешнему эталону поведения** без операционализации — замечание к Issue с требованием явно перечислить контрольные шаги или выражения или сослаться на связанные Issue со сценариями (`#N`). Критерий: требование готово к разработке, если метод проверки **не требует** догадки «снять с эталона на месте».

## Сводка: свойства требований vs состояние системы

| Свойство (IEEE 830) | Ожидание в продукте до эксперимента | Что показал эксперимент |
|---------------------|-------------------------------------|-------------------------|
| Однозначность | Верификация Issue, локально | В основном достижимо для текста Issue |
| Полнота (отдельная Issue) | Верификация Issue | Слабо для визуальных и «как эталон X» требований без вложений |
| Полнота (совокупность) | Человеческий процесс | Разрыв усиливается при плоском списке Issue |
| Непротиворечивость | Верификация Issue, локально + `#N` | Работает для текста; структура не помогает при каскадных правках |
| Проверяемость | Верификация Issue | Текст может быть «проверяемым» при непроверяемом эталоне |
| Приоритизация | Не в scope продукта | В эксперименте задаётся в спецификации; в Issue — вручную |
| **Модифицируемость** | **Побочный эффект** | **Не достигается; инфраструктурный разрыв (F-001, F-004)** |
| Отслеживаемость | Перспектива развития | Upstream есть в спецификации; в GitLab — только `#N`, без артефактов |

## Как пополнять этот документ

По мере прохождения эксперимента (этапы 3–4 и последующие итерации) добавляются новые находки по шаблону:

1. **Идентификатор** (`F-00N`).
2. **Проблема** — что не работает или работает не так, как предполагалось.
3. **Проявление в эксперименте** — конкретный эпизод, без отсылок к файлам других репозиториев.
4. **На какие свойства влияет**.
5. **Желательный вид решения** — какие способности должна получить система; без выбора технологии и без детального дизайна.

Документ ведётся в проекте эксперимента и **переносится** в репозиторий GitLab AI Helper для работы ИИ-агента над тем продуктом.
