# FINDINGS — выводы эксперимента для развития GitLab AI Helper

Документ предназначен для **ИИ-агента**, который дорабатывает проект **GitLab AI Helper** — backend-сервис для верификации Issue и Merge Request в GitLab с помощью ИИ, включая MCP-сервер для чтения и изменения требований (Issue) из облачной LLM.

Здесь зафиксированы проблемы, обнаруженные в ходе эксперимента по No-/Low-coding разработке ПО: путь от пожеланий заказчика через роли «аккаунт-менеджер» и «бизнес-аналитик» до спецификации требований и переноса их в GitLab Issue. **Конкретные решения в документе не предлагаются** — только описание разрывов и ожидаемый **вид** того, чего не хватает в системе.

Документ **самодостаточен**: для понимания контекста не требуются другие файлы.

---

## Контекст эксперимента (кратко)

Эксперимент воспроизводит цепочку ролей: сбор информации от заказчика → бизнес-анализ и спецификация требований → постановка задач разработчику → разработка.

На этапе бизнес-анализа получена спецификация из 29 пронумерованных требований (`REQ-001` … `REQ-029`) к сетевому калькулятору, включая:

- программный интерфейс (API) и веб-страницу режима Basic;
- эталон поведения Ubuntu 24.04 Basic;
- визуальные эталоны (скриншоты: основной вид, hover, press, ошибки);
- сценарии приёмки и критерии готовности v1.

Пользователь переносит требования и задачи в GitLab Issue **вручную** или через MCP-сервер GitLab AI Helper. В ходе переноса выяснилось: система **не умеет работать с вложениями (uploads)** в Issue. Визуальные эталоны пришлось выложить во внешний репозиторий; в тексты требований вставлены прямые URL на raw-файлы, чтобы отдельное требование оставалось **полным** даже при изъятии из общего списка.

В описании GitLab AI Helper свойство **модифицируемости** требований (IEEE 830: связная структура, оглавление, перекрёстные ссылки, раздельное выражение каждого требования) считалось **автоматически достижимым** как побочный эффект проверок однозначности, полноты, непротиворечивости и проверяемости отдельной Issue. Эксперимент показал, что это предположение **неверно**.

На этапе постановки задач (продукт-менеджер) выяснилось: результаты верификации Issue публикуются как комментарии бота в GitLab, но **недоступны** ИИ-ассистенту через MCP — пользователь вынужден вручную копировать markdown из формы редактирования комментария.

## F-001. Модифицируемость ошибочно принята за побочный эффект

### Проблема

Модифицируемость **совокупности** требований не возникает сама по себе от качества текста отдельных Issue. При переносе единой спецификации в **однородный список Issue** без типизации теряются оглавление, группы, иерархия и единая картина зависимостей.

### Проявление в эксперименте

- Спецификация имела оглавление, девять групп требований (G1–G9), перекрёстные ссылки `REQ-…`, приложения со сценариями проверки и критериями приёмки релиза.
- После разнесения по отдельным Issue структура **исчезла**: каждая Issue видна изолированно.
- Правка одного требования (например, текста ошибки для квадратного корня из отрицательного числа) не сопровождается автоматическим указанием на **зависимые** Issue; связи `#N` в тексте есть, но **оглавления, указателя и обзора корпуса** нет.

### На какие свойства влияет

- **Модифицируемость** — основной удар.
- **Непротиворечивость** (глобальная) — при правке одного требования сложнее увидеть каскад по всей совокупности.
- **Полнота совокупности** — риск дублирования и расхождения формулировок при ручном разнесении.

### Желательный вид решения (без проектирования)

Система должна **явно** поддерживать модифицируемость совокупности требований, а не полагаться только на качество отдельной Issue. Ожидаемые способности:

- **обзор** корпуса требований (список, фильтры, группировка — не только знание номера Issue заранее);
- сохранение или восстановление **структуры** (уровни, группы, родитель–потомок) при хранении в GitLab;
- при изменении Issue — **сигнал** о затронутых связанных требованиях (по `#N` и, по возможности, по смыслу);
- различение ролей Issue (требование vs тикет), чтобы легитимные различия уровней не считались противоречием.

## F-002. Вложения Issue не участвуют в жизненном цикле требования

### Проблема

GitLab AI Helper оперирует **только текстом** Issue (заголовок, описание) и историей версий текста. **Вложения** (файлы, загруженные в Issue в GitLab) не учитываются при верификации Issue, не доступны через MCP и не попадают в локальный снимок требования.

Для требований, где эталон приёмки — **изображение** (макет, скриншот), это создаёт **инфраструктурный разрыв**.

### Проявление в эксперименте

- Бизнес-аналитик оформил визуальные требования (совпадение цветов и размеров, реакция на hover и press, оформление ошибки деления на ноль) со ссылками на эталонные PNG.
- Перенести изображения **внутрь** GitLab Issue через MCP **невозможно** — функционал uploads отсутствует.
- Обходной путь: внешний хостинг файлов + raw-URL в markdown-тексте Issue. Это компенсация ограничения системы, а не её свойство.

### На какие свойства влияет

- **Полнота отдельной Issue** — сценарий верификации Issue формально проверяет критерии приёмки в тексте, но **не может** убедиться, что визуальный эталон доступен вместе с требованием.
- **Проверяемость** — критерий «сравнить с эталонным скриншотом» без вложения или устойчивой внутренней ссылки **неисполним** из одного артефакта GitLab.
- **Модифицируемость** — при смене эталона нужно править URL во многих Issue; версионирование изображений не связано с историей Issue.
- **Отслеживаемость** — связь «требование → эталон-изображение» вынесена за пределы системы.

### Желательный вид решения (без проектирования)

Требование в системе должно представлять **текст + связанные артефакты** как единое целое. Ожидается:

- **приём событий** GitLab о добавлении, замене и удалении вложений Issue и сохранение метаданных (и при необходимости содержимого) в том же контуре, что и текст;
- **MCP-инструменты**: перечислить вложения требования, прочитать или скачать файл (с учётом типа и размера), при необходимости — загрузить или заменить вложение в Issue;
- **верификация Issue**: учитывать вложения при оценке полноты и проверяемости (визуальное требование без прикреплённого или явно связанного эталона — замечание к Issue).

## F-003. Самодостаточность Issue при изъятии из спецификации

### Проблема

При переносе требований из спецификации в отдельные Issue предполагается, что **каждая Issue самодостаточна**: из её текста понятны поведение, критерии проверки и контекст **без** чтения всего документа-спецификации. На практике самодостаточность **неравномерна**.

### Проявление в эксперименте

| Категория требований | Самодостаточность при переносе в одну Issue |
|----------------------|---------------------------------------------|
| API, изоляция пользователей, состояние | Высокая — текст достаточен |
| Визуальные (внешний вид, hover, press, ошибки) | Только при явных URL на внешние файлы |
| Поведенческие «как Ubuntu 24.04 Basic» | Низкая — нужен живой эталон или полный перечень сценариев внутри Issue |

Сценарий верификации Issue проверяет полноту **текста**, но **не требует**, чтобы внешние эталоны были **доступны из Issue** (вложение, артефакт в контуре GitLab, устойчивый идентификатор).

### На какие свойства влияет

- **Полнота** (отдельной Issue) — формально может быть удовлетворена при пустой отсылке «как в эталоне».
- **Проверяемость** — исполнитель и приёмщик не могут проверить требование, имея только Issue.
- **Корректность** — риск расхождения между формулировкой в Issue и эталоном вне системы.

### Желательный вид решения (без проектирования)

Верификация и MCP должны **различать** отсылки, которые делают требование проверяемым, и отсылки-заглушки. Желательно:

- правило или проверка: если критерий приёмки ссылается на файл, изображение или внешний эталон — артефакт **присутствует** как вложение Issue или как однозначно разрешаемая ссылка в контуре проекта;
- подсказка при создании или редактировании требования: что включить в текст или вложения для самодостаточности;
- для поведенческих эталонов — явное требование перечислить **контрольные сценарии** в теле Issue или связать Issue со сценарием через `#N`, а не отсылать к «сравнению с продуктом X» без операционализации.

## F-004. Разрыв между монолитной спецификацией и однородным списком Issue

### Проблема

Спецификация требований по IEEE 830 — **единый документ** с введением, границами версии, допущениями, приложениями. GitLab AI Helper опирается на модель **одной Issue = один текстовый объект** без типизации. Перенос из документа в список Issue **не сохраняет** метаданные уровня документа.

### Проявление в эксперименте

- Решения анализа (стиль API, механизм идентификации клиента, модель операций) **намеренно не** входят в пронумерованные требования, но нужны продукт-менеджеру и разработчику.
- Критерии приёмки релиза (пять блоков AC) и приложение со сценариями — **вне** отдельных Issue; при разнесении только REQ теряется контекст приёмки v1.
- Пользователь вынужден дублировать структуру вручную или держать «мастер-документ» вне GitLab.

### На какие свойства влияет

- **Модифицируемость** — нет единой точки входа в совокупность требований.
- **Полнота совокупности** — система не помогает сверить «все ли требования из спецификации заведены».
- **Отслеживаемость** — цепочка «спецификация → Issue» не зафиксирована в системе.

### Желательный вид решения (без проектирования)

Возможность **не сводить всё к плоскому списку** без потери структуры:

- типизация или метаданные Issue (требование / тикет / критерий приёмки / решение анализа / приложение);
- Issue-«контейнер» или эквивалент для границ версии и общих критериев приёмки со ссылками на дочерние `#N`;
- инструмент сверки: ожидаемый набор идентификаторов требований vs фактически заведённые Issue.

## F-005. Внешние URL как обходной путь ломают доверие к хранилищу

### Проблема

Вставка raw-URL на файлы во внешнем репозитории в текст Issue **имитирует** полноту требования, но создаёт зависимости: доступность внешнего хоста, неизменность ветки и пути, права доступа, отсутствие версионирования вместе с историей Issue.

### Проявление в эксперименте

- Для визуальной приёмки в Issue вставлены ссылки на эталонные PNG (основной вид калькулятора, hover, press, ошибка деления на ноль).
- Требование **формально полное** в тексте, но **фактически** эталон не принадлежит GitLab-проекту требований.
- Смена эталона или удаление внешнего репозитория **ломает** проверяемость без изменения текста Issue.

### На какие свойства влияет

- **Проверяемость**, **полнота**, **отслеживаемость**, **модифицируемость** — все через внешнюю зависимость.

### Желательный вид решения (без проектирования)

Система должна предпочитать **артефакты в контуре GitLab** (вложения Issue, файлы в репозитории проекта с явной связью в тексте). При обнаружении критичных внешних URL в критериях приёмки — **предупреждение** при верификации Issue. Критерий готовности: визуальное требование полно, если эталон **доступен из того же проекта**, что и Issue.

## F-006. MCP не закрывает разрыв переноса спецификации

### Проблема

MCP-сервер умеет читать, создавать и изменять **текст** Issue и смотреть историю. Перенос 29 требований из подготовленных текстов для системы хранения остаётся **ручным** или полуручным: нет пакетного создания, нет привязки вложений, нет восстановления структуры групп и связей «критерий приёмки ↔ сценарий ↔ REQ».

### Проявление в эксперименте

- Пользователь копирует тексты в Issue по одному.
- Связи `#N` между требованиями выставляются вручную после создания всех Issue.
- Перенос **трудоёмок** и источник ошибок (пропуск требования, неверная ссылка).

### На какие свойства влияет

- **Модифицируемость** — ошибки при ручном копировании.
- **Непротиворечивость** — неверные или отсутствующие `#N`.
- **Полнота совокупности** — пропуск отдельных REQ.

### Желательный вид решения (без проектирования)

Возможности, снижающие ручной труд при **импорте** структурированной спецификации в GitLab (точный дизайн не задаётся):

- создание нескольких Issue из структурированного ввода с сохранением идентификаторов (`REQ-001` …) в заголовке или метаданных;
- отложенная простановка связей `#N` после создания корпуса;
- загрузка вложений вместе с текстом требования;
- отчёт о расхождениях между ожидаемым набором и созданными Issue.

## F-007. Верификация Issue не учитывает зависимость от живого внешнего эталона поведения

### Проблема

Часть требований формулируется как «поведение совпадает с Ubuntu 24.04 Basic»: проверка предполагает **внешнюю программу** и пошаговое сравнение. Сценарий верификации Issue оценивает формулировку в тексте, но **не фиксирует**, что для приёмки нужен внешний эталон и перечень контрольных шагов **внутри** требования или связанных Issue.

### Проявление в эксперименте

- Требование о совпадении промежуточных состояний с эталоном: конкретные числа для многих операций не перечислены в Issue — они снимаются с Ubuntu при прогоне (допущение аналитика).
- Issue с текстом «как в Ubuntu» может пройти проверку полноты, оставаясь **непригодной** для разработчика без доступа к эталону и сценариям.

### На какие свойства влияет

- **Проверяемость**, **однозначность** (на практике), **полнота** отдельного требования.

### Желательный вид решения (без проектирования)

При обнаружении отсылки к **именованному внешнему эталону поведения** без операционализации — замечание к Issue с требованием явно перечислить контрольные шаги или выражения или сослаться на связанные Issue со сценариями (`#N`). Критерий: требование готово к разработке, если метод проверки **не требует** догадки «снять с эталона на месте».

## F-008. Результаты верификации Issue недоступны ИИ-ассистенту через MCP

### Проблема

Система верификации Issue пишет выводы (однозначность, полнота, непротиворечивость, проверяемость, вердикт о готовности к разработке) в **комментарий** к Issue в GitLab. MCP-сервер умеет читать и изменять **текст** Issue (заголовок, описание) и историю версий текста, но **не отдаёт** ИИ-ассистенту результаты верификации и связанные комментарии бота.

Из‑за этого роль, которой нужен разбор замечаний (в эксперименте — **продукт-менеджер** после переноса задач в Issue), **не может** получить вывод верификатора из того же контура, где живёт требование/задача. Цепочка «Issue создана → верифицирована → обсуждение с Пользователем → правка» ломается на шаге чтения результата: без участия человека-копировщика ассистент «слеп».

### Проявление в эксперименте

- После постановки задач первой волны Пользователь прогонял верификацию Issue (в том числе повторно после переформулирования).
- Чтобы передать вывод продукт-менеджеру (ИИ-ассистенту), Пользователю приходилось: открыть Issue → найти комментарий бота → Edit Comment → режим **plain editing** → скопировать markdown вручную в чат.
- MCP `load-requirement` / история версий давали только описание Issue, **без** структурированного или текстового отчёта верификации.

### На какие свойства влияет

Прямо не свойство IEEE 830 у текста требования, а **замкнутость контура** верификации и сопровождения:

- **Модифицируемость** на практике — правка формулировки после замечаний зависит от ручного моста «GitLab UI → чат».
- **Проверяемость процесса** — нельзя воспроизвести обсуждение замечаний, опираясь только на MCP и Issue.
- Скорость итераций «верификация → переформулирование → повторная верификация» падает; риск потери или искажения текста при копировании.

### Желательный вид решения (без проектирования)

ИИ-ассистент (в том числе в роли продукт-менеджера) должен **читать выводы верификации** той же Issue, не требуя ручного копирования из UI GitLab. Ожидаемые способности:

- получить **последний** (и при необходимости **исторические**) отчёты верификации Issue: разбор по свойствам, замечания, вердикт о готовности к разработке;
- доступ через MCP (или эквивалентный программный интерфейс) по номеру Issue / идентификатору требования;
- текст отчёта в пригодном для обсуждения виде (например markdown или структурированные поля), совпадающий с тем, что пользователь видит у бота;
- опционально — список или фильтр комментариев/событий верификации, чтобы отличать отчёт бота от обычных обсуждений.

Критерий: Пользователь говорит ассистенту «разбери верификацию #N» — ассистент загружает вывод **сам**, без Edit Comment и plain editing.

## F-009. Верификация связанных Issue конфликтует с обязательной трассировкой

### Проблема

Текущая версия GitLab AI Helper при верификации Issue **находит в тексте ссылки** на связанные требования и задачи и анализирует **текущую** Issue **относительно них** (обход связей в глубину, например до **трёх** уровней: текущая → `#1` → `#2` → `#3`, совокупность рассматривается вместе).

Отсюда практическое противоречие процесса:

- **если не указывать** связанные `#N` / `REQ-…` — замечаний и «конфликтов» с соседними формулировками меньше, вердикт пройти проще;
- **не указывать связанные требования нельзя** — трассируемость и осмысленная постановка задач как раз требуют ссылок на `REQ-… #<номер>` (и при необходимости на другие Issue).

Система одновременно **поощряет** изоляцию текста (меньше связей → легче «зелёный» вердикт) и **предполагает** связный корпус требований. Исполнитель процесса оказывается в ловушке: соблюдение отслеживаемости ухудшает шансы пройти ту же верификацию.

Дополнительно всплывает смешение **ролей** Issue: инфраструктурная задача (подготовка окружения, осознанная недоопределённость технологий «на выходе в `tech-stack.md`») проверяется так же, как продуктовое требование. Ссылки на связанные `REQ-…` подтягивают в разбор ожидания уровня API, форматов состояния, критериев приёмки продукта — хотя задача их **намеренно** не закрывает.

### Проявление в эксперименте

- На этапе постановки задач Продукт-менеджер обязан трассировать задачи к `REQ-… #<номер>` (в том числе с ролью «не нарушить»).
- Задача Ведущему разработчику (#57, подготовка репозитория и окружения): в тексте явно отложены механизм привязки клиента к состоянию, детали API и т.п. — их должен выбрать Ведущий и зафиксировать в `tech-stack.md`; продуктовые сценарии — в следующих задачах Разработчика.
- Повторные прогоны верификации (в т.ч. третий вердикт) завершались **«не готово»**: недостаёт как раз отложенных деталей; полнота/проверяемость оцениваются с опорой на связанные требования и внешние отсылки; при этом внутренние противоречия в самой Issue не находились.
- Наблюдение экспериментатора: при отсутствии ссылок на связанные Issue конфликтов было бы меньше — но опускать трассировку недопустимо.
- На этом эпизоде результаты верификации **проигнорированы** (задача передаётся дальше без доведения вердикта до «готово»); решение, что менять в продукте Helper, **отложено**.

### На какие свойства влияет

- **Отслеживаемость** — формально нужна; фактически наказывается верификатором через разбор связанных Issue.
- **Непротиворечивость** (как её считает система) — растёт число «касаний» с чужим уровнем детализации при каждом `#N`.
- **Полнота** и **однозначность** отдельной Issue — к инфраструктурной задаче применяются ожидания полноты продуктовых `REQ-…` из окрестности ссылок.
- **Модифицируемость** процесса — стимул искусственно обеднять текст связями, чтобы пройти проверку.

### Желательный вид решения (без проектирования)

**Пока не формулируется полностью.** Зафиксирована только ситуация и противоречие «обязательная трассировка ↔ верификация с обходом связанных Issue». Конкретный вид изменений в GitLab AI Helper (учёт роли Issue, глубины обхода, профиля проверки для задач vs требований и т.п.) — **решим позже**.

**Дополнение (2026-07-18):** правка промпта верификатора Issue (см. **F-010**) частично сужает обход связей и легитимирует границы/ссылки; после неё корпус требований и задач прошёл верификацию с положительным вердиктом. Это **не** закрывает F-009 как продуктовую тему целиком — эпизод с инфраструктурной задачей и смешением ролей Issue остаётся в истории эксперимента.

## F-010. Промпт верификатора Issue требует «самодостаточный тест-спек» вместо поведенческого требования

### Проблема

Промпт (и фактическое поведение) верификатора Issue в GitLab AI Helper заточен так, будто каждая Issue должна быть **самодостаточным техническим тест-спеком**: со всеми входами/выходами, протоколом, порогами, пошаговым тест-планом и без опоры на связанные требования. Это **расходится** с инженерией требований и с инструкцией бизнес-аналитика эксперимента: в `REQ-…` — поведение и наблюдаемые свойства; детали реализации и осознанно незаданное — **вне** требования; смежная ответственность закрывается **ссылками** `#N`.

Из‑за этого корректно сформулированные поведенческие требования систематически получают отрицательный вердикт: верификатор штрафует за границы роли BA, за трассируемость и за метод проверки без детального тест-кейса.

### Проявление в эксперименте

После превентивной доработки формулировок всего корпуса требований (инструкция BA) четыре требования (**REQ-004**, **REQ-016**, **REQ-018**, **REQ-019**) при прежнем промпте оставались «не готовы». Совместный вывод экспериментатора и BA: при текущем промпте пройти верификацию для них **невозможно**, не нарушая границы `REQ-…`.

Типичные ошибочные штрафы:

1. **Нет понятия «границы требования» / «намеренно не задано».** Явное «не задаётся» (протокол, механизм идентификации, RPS/SLA, формат хранения и т.п.) трактуется как пробел полноты, хотя по роли BA это детали реализации.
2. **Полнота = самодостаточность без ссылок.** Отсылки к принятым связанным Issue (`#N`) и сценариям приёмки штрафуются («не проверить без #9 / #14»), хотя загрузка связанных предписана, а трассируемость — норма процесса.
3. **Проверяемость дрейфует в тест-план.** Требуются пошаговые кейсы, числовые пороги, имитация окружения; экспертная/визуальная сверка с эталоном без пиксельного допуска считается дефектом.
4. **Антиконфликтная установка порождает ложные конфликты.** «Не смягчать» побуждает называть несуществующие противоречия (например «открытый доступ vs аутентификация», «cookie не задаётся vs нужна идентификация»).
5. **Глубокий обход связей + путаница ID.** Обход до нескольких уровней раздувает контекст; замечания с соседних Issue переносятся на проверяемую; путаются номера требований.
6. **Нет порога вердикта.** Любая косметическая придирка ведёт к «требуется доработка»; не разделено «блокер свойства» vs «необязательное замечание».

После переработки промпта верификатора Issue в GitLab AI Helper **все** требования и **все** задачи экспериментального корпуса получили **положительный** вердикт. Отдельный анализ промпта верификации Merge Request в этой итерации **не принят** (вывод экспериментатору не понравился; правки MR отложены).

### На какие свойства влияет

- **Полнота** (как её считает система) — ложные пробелы из‑за границ и ссылок.
- **Проверяемость** — завышение до тест-плана.
- **Непротиворечивость** — ложные конфликты и шум от глубокого обхода связей.
- **Однозначность** — побочно (расфокус на чужие Issue и чужие ID).
- **Отслеживаемость** — пересечение с **F-009**: ссылки нужны процессу, но при старом промпте ухудшали вердикт.

### Желательный вид решения (без проектирования)

Верификатор Issue (промпт и критерии вердикта) должен уметь оценивать **поведенческое** требование / задачу в границах роли автора текста. Ожидаемые способности:

- признавать **осознанную границу** («не задаётся» / вне объёма наблюдаемого поведения) **не** пробелом полноты; не требовать деталей внутреннего устройства, протокола, SLA и т.п., если они не относятся к заявленному поведению;
- считать ссылку на связанное требование/задачу (`#N`) **допустимым и достаточным** способом закрыть смежную ответственность; не требовать самодостаточности «без ссылок» при загруженных связанных Issue;
- считать достаточным **понятный метод** проверки и наблюдаемый результат; не требовать пошагового тест-плана, числовых порогов и имитации окружения; экспертное/визуальное сравнение с эталоном — валидный метод без обязательного числового допуска;
- называть конфликт только при **фактической** невозможности выполнить формулировки одновременно; отсутствие детали реализации и ссылка на смежное требование конфликтом не являются;
- ограничивать разбор связей **прямыми** ссылками (без глубокого переноса замечаний с дальних узлов); все замечания — только про проверяемую Issue;
- выносить отрицательный вердикт только при **блокере** одного из четырёх свойств в границах Issue; косметику помечать отдельно и не использовать как основание «не готово».

Критерий: поведенческие требования с явными границами и трассировкой `#N` (как REQ-004 / 016 / 018 / 019 в эксперименте) проходят верификацию без вынужденного внесения деталей реализации в текст `REQ-…`.

## F-011. Не догонять CLI GitLab: оставить уникальное для Helper, остальное — экосистеме

### Проблема

GitLab AI Helper (в т.ч. MCP) частично пересекается с тем, что уже умеют официальные и общепринятые клиенты GitLab (`glab` и аналоги) и сам GitLab. Попытка реализовать «весь» CLI/API-поверхностный функционал этих утилит нереалистична и отвлекает от ценности продукта. В эксперименте ИИ-ассистент Разработчика тянулся к `glab`/`gh` вместо MCP Helper — это и симптом пробела привычных операций, и соблазн раздувать Helper до замены CLI.

### Проявление в эксперименте

- На этапе разработки агент пытался вызывать `glab` / `gh` для операций вокруг Issue и Merge Request, хотя контракт эксперимента для Issue — MCP Helper.
- Создание MR разными моделями через CLI давало нестабильный результат; процесс эксперимента сознательно оставил создание MR человеку, а Issue — MCP.
- Параллельно ясно: полностью воспроизвести возможности `glab` в Helper **не нужно** и вряд ли получится.

### На какие свойства влияет

Прямо не свойство IEEE 830 отдельной Issue, а **границы продукта** и сценарий обеспечения свойств через инструмент:

- доступ ИИ к **истории изменений** требования (уникальная для Helper способность относительно «голого» GitLab UI/API в текущей модели эксперимента);
- ясность, что верификация и работа с формулировками требований — зона Helper, а рутинный GitLab workflow — зона экосистемы GitLab;
- долгосрочная **модифицируемость** и размещение корпуса требований (перспектива Wiki).

### Желательный вид решения (без проектирования)

- **Не** ставить целью паритет с `glab` / `gh`.
- **Отказаться** от части собственного функционала Helper там, где операция уже надёжно закрывается экосистемой GitLab / `glab`; направлять пользователей и агентов туда, а не дублировать.
- **Сохранить и развивать** в Helper/MCP только то, чего **нет** (или недостаточно) в самом GitLab для сценария требований — на момент эксперимента образец: **история изменений Issue** (версии формулировок) и связанная с верификацией/свойствами требований работа.
- В **далёкой перспективе** — перенос и ведение требований в **Wiki проекта** (как контур хранения корпуса), без требования спроектировать это сейчас.

Критерий направления: дорожная карта Helper сужается к уникальной ценности для требований/верификации; регрессии «ещё один неполный gitLab CLI» избегают.

## F-012. Лимит TPM провайдера блокирует верификацию Issue; повтор сразу не помогает

### Проблема

Верификация Issue зависит от внешнего LLM-провайдера и его **лимита токенов в минуту (TPM)** на конкретную модель. При исчерпании лимита проверка **не выполняется**, хотя формулировка Issue может быть в порядке. Повторный запуск вскоре после ошибки часто снова упирается в тот же TPM (окно лимита ещё не восстановилось или тот же тяжёлый запрос снова не помещается).

Система при этом может вести себя корректно: писать в комментарий к Issue текст ошибки провайдера, а не молчаливый сбой и не ложный вердикт о качестве требования/задачи.

### Проявление в эксперименте

- При верификации задачи `#39` дважды подряд получен отказ провайдера вида: *Rate limit exceeded* для модели **gpt-5-nano**, лимит **200 000 TPM**, уже использовано ~167k, запрошено ещё ~63k (не хватило окна); рекомендация провайдера — повторить через ~9 с.
- Оба повтора дали ту же ошибку.
- GitLab AI Helper **отработал верно**: к Issue добавлен комментарий с сообщением об ошибке (не сфабрикованный вердикт по свойствам Issue).
- Фон по стоимости на **gpt-5-nano** за всё время использования в организации эксперимента: порядка **$0.58** при ~**5.3M** total tokens и **556** запросах — то есть отказ связан не с «дорогим» объёмом трат, а с **минутным** потолком TPM на nano при тяжёлом запросе (типично — Issue плюс связанные по `#N`).

### На какие свойства влияет

Не свойство IEEE 830 текста Issue, а **доступность сценария верификации**:

- однозначность / полнота / непротиворечивость / проверяемость **не оцениваются**, пока запрос к модели не проходит;
- ложное впечатление, что «задача плохая», если оператор не отличает ошибку провайдера от отрицательного вердикта;
- итерации постановки и волны задач **блокируются** инфраструктурой, а не качеством формулировок.

### Желательный вид решения (без проектирования)

- Сохранять (и не ухудшать) поведение: ошибка провайдера / rate limit — **явный комментарий** к Issue, без подмены вердиктом о свойствах.
- Уметь **отличать** для оператора и для ИИ-ассистента (в т.ч. через будущий доступ к отчётам верификации, см. F-008) инфраструктурный отказ от содержательного «не готово».
- Иметь стратегию устойчивости к TPM: отложенный повтор с учётом окна лимита, очередь, уменьшение объёма контекста связанных Issue, выбор модели/квоты с достаточным TPM под реальный размер прогона — **без** требования зафиксировать конкретную модель в этом документе.
- Не считать низкую суммарную стоимость использования доказательством, что TPM-лимит «не должен срабатывать»: минутный потолок и накопленный spend — разные оси.

Критерий: повторная верификация после transient rate limit либо проходит, либо снова даёт **понятный** инфраструктурный отказ; содержательный вердикт появляется только после успешного ответа модели.

## F-013. Семантический поиск требований: сначала готовый OpenAI Vector Store, затем своё хранилище

### Проблема

Непротиворечивость и «ненарушение» опираются на поиск **релевантных** требований по смыслу, а не только по явным ссылкам `#N`. Без семантического поиска верификатор и постановщики задач видят лишь локальный граф связей; среди сотен и тысяч Issue легко не найти смысловой конфликт или ограничение. В продукте GitLab AI Helper такой поиск **ещё не готов**; строить с нуля собственное хранилище эмбеддингов до появления рабочего сценария поиска — лишняя задержка.

### Проявление в эксперименте

- В модели требований эксперимента (и в постановке Helper) уже зафиксировано ограничение: обход только по `#N`, без поиска по смыслу — глобальная непротиворечивость и отслеживаемость «ненарушения» недостижимы в полном виде.
- Инструкции ролей (в т.ч. реакция BA на верификацию) ранее откладывались в ожидании семантического поиска в Helper — инструментарий так и не появился к этапам постановки и разработки.
- У провайдера, которым уже пользуется верификация (OpenAI), есть готовый контур **Vector stores / Retrieval API**: загрузка текстов → семантический `search` по индексу. В консоли платформы это видно как отдельный ресурс (vector stores) — кандидат на быстрый пилот без собственного векторного бэкенда.

### На какие свойства влияет

- **Непротиворечивость** — поиск кандидатов на конфликт за пределами `#N`.
- **Отслеживаемость** / сценарий **ненарушения** — подбор релевантных требований к тикету или MR по смыслу.
- Косвенно **модифицируемость** — при правке требования проще увидеть смысловой каскад, не только явные ссылки.

### Желательный вид решения (без проектирования)

Двухфазный подход к хранению индекса для семантического поиска:

1. **Сначала** — опереться на **уже готовое** решение OpenAI (vector store + retrieval): научиться синхронизировать тексты требований/задач из GitLab, выполнять поиск по смыслу в сценариях верификации и помощи ролям. Цель фазы — получить рабочий функционал поиска в Helper и проверить ценность на реальном корпусе, а не спроектировать идеальное хранилище.
2. **Потом**, когда функционал семантического поиска в GitLab AI Helper **уже готов и обжит** — **переключиться** на **своё** решение хранения таких данных (свой индекс/эмбеддинги/хранилище — конкретный выбор вне этого документа), сохранив тот же продуктовый контракт поиска для верификатора и MCP.

Критерий: поиск по смыслу доступен в Helper до завершения «своего» хранилища; смена бэкенда индекса не ломает сценарии, уже завязанные на семантический поиск; явные `#N` остаются, семантика их дополняет.

## F-014. Ошибка UTF-8 в JSON на уровне Symfony AI при верификации

### Проблема

Запрос верификации (или связанный вызов к LLM через стек Symfony AI) может падать с ошибкой вида: *Invalid value for "json" option: Malformed UTF-8 characters, possibly incorrectly encoded*. Сбой проявляется **ниже** прикладного кода GitLab AI Helper — в макете/пакетах **Symfony AI**, ещё до стабильного релиза этих компонентов. Тексты требований и задач на русском и с символами вроде **π** (пи) для продукта нормальны; инфраструктура кодирования JSON/UTF-8 не должна на них ломаться.

### Проявление в эксперименте

- В ходе работы с Helper получена указанная ошибка при обработке запроса, содержащего (или способного содержать) UTF-8 вне базового ASCII.
- Диагностика указывает на слой Symfony AI (макет пакетов), а не на ошибку формулировки Issue или логику вердикта Helper на «своём» уровне.
- До стабильного релиза пакетов Symfony AI полноценный фикс на стороне Helper может быть преждевременным или невозможным без обновления зависимости.

### На какие свойства влияет

Снова **доступность** сценария верификации (как F-012), а не качество текста Issue:

- проверка не доходит до содержательного разбора свойств;
- риск систематических отказов на корпусах с кириллицей, формулами, символом π и прочим Unicode.

### Желательный вид решения (без проектирования)

- После **первого стабильного релиза** пакетов Symfony AI, на которых строится Helper, — **обязательный регрессионный прогон**: запросы верификации с корректным UTF-8, включая кириллицу и отдельные символы вроде **π** (пи), не должны давать *Malformed UTF-8* / ошибку опции `json`.
- До стабильного релиза — фиксировать такие отказы как инфраструктурные (явный комментарий/лог), не трактовать как брак требования.
- При сохранении ошибки после стабильного релиза — дефект зависимости или обёртки; устранять на соответствующем уровне, не «вычищая» Unicode из текстов требований.

Критерий: верификация Issue с π и кириллицей в тексте проходит без ошибки кодирования JSON/UTF-8 на актуальном стабильном Symfony AI.

## Сводка: свойства требований vs состояние системы

| Свойство (IEEE 830) | Ожидание в продукте до эксперимента | Что показал эксперимент |
|---------------------|-------------------------------------|-------------------------|
| Однозначность | Верификация Issue, локально | В основном достижимо для текста Issue; при старом промпте — шум от чужих ID (**F-010**) |
| Полнота (отдельная Issue) | Верификация Issue | Слабо для визуальных и «как эталон X» требований без вложений; старый промпт путал границы с пробелами (**F-010**) |
| Полнота (совокупность) | Человеческий процесс | Разрыв усиливается при плоском списке Issue |
| Непротиворечивость | Верификация Issue, локально + `#N` | Работает для текста; структура не помогает при каскадных правках; обход связанных Issue углублял конфликт с трассировкой (**F-009**); смягчение в промпте — **F-010**; без семантики — потолок (**F-013**) |
| Проверяемость | Верификация Issue | Текст может быть «проверяемым» при непроверяемом эталоне; старый промпт завышал до тест-плана (**F-010**) |
| Приоритизация | Не в scope продукта | В эксперименте задаётся в спецификации; в Issue — вручную |
| **Модифицируемость** | **Побочный эффект** | **Не достигается; инфраструктурный разрыв (F-001, F-004)** |
| Отслеживаемость | Перспектива развития | Upstream есть в спецификации; в GitLab — только `#N`; **F-009** — ссылки обязательны процессу, но ухудшают прохождение верификации; смысловой подбор — **F-013** |
