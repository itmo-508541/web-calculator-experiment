# Эксперимент

Хочу поставить эксперимент про разработку программного обеспечения. В эксперименте будем участвовать мы с тобой (то есть пользователь и ИИ-ассистент). На каждом этапе мы будем выполнять разные роли.

В данном проекте наша роль — наблюдателей и незримых помощников всего процесса. Мы формулируем и ставим задачи для последующих этапов, анализируем результаты. Ты составляешь инструкции для ИИ-ассистентов на каждом этапе. В конце эксперимента обсудим результаты и сформулируем выводы.

## Цель эксперимента

Эксперимент реализует **No- и/или Low-coding подходы в разработке программного обеспечения**. В рамках эксперимента нужно понять, воспроизвести и описать аспекты **планирования** разработки ПО; на основе планирования — проектирование и разработка.

Параллельная цель (связанная с другим проектом): предложить **простой сценарий обеспечения наличия каждого свойства требования** — см. [REQUIREMENT.md](REQUIREMENT.md).

Процесс **многоитерационный**: в любой момент могут поступить новые пожелания; на стадии разработки может выясниться невозможность или нецелесообразность требований; может обнаружиться неполнота спецификации — и тогда нужно возвращаться к общению с заказчиком.

## Роли в процессе

| # | Роль | Назначение |
| --- | --- | --- |
| 1 | **Аккаунт-менеджер** | Сбор информации от заказчика, проведение интервью |
| 2 | **Бизнес-аналитик** | Анализ собранной информации, подготовка спецификации требований |
| 3 | **Продукт-менеджер** | Формирование задач для разработчиков, управление приоритетами |
| 4 | **Разработчик** | Создание программного обеспечения |

На этапе разработки пользователь выступает как **вайб-кодер** — участник без навыков программирования, у которого уже есть задачи и приоритеты. Цель предыдущих этапов — обеспечить ему возможность участвовать в разработке ПО. Возможен предварительный шаг «настройки окружения» (senior-разработчик помогает вайб-кодеру) — TBD.

**Заказчик** — отдельная роль на первом шаге; пользователь играет её при работе с Аккаунт-менеджером. Роль Заказчика отдельно не логируется — вместо этого фиксируется артефакт работы Аккаунт-менеджера.

## Логирование действий экспериментатора

ПО для автоматического добавления и связывания требований и задач в GitLab **пока не готово** — на этапах 2 и 3 пользователь выполняет эти действия **вручную** (при необходимости с помощью MCP-сервера `gitlab-ai-helper`). Действия пользователя на этапах, где он выполняет роль **Пользователя** (этапы 2–3), а также в роли **Вайб-кодера** (этап 4), **нужно логировать** для анализа эксперимента после его завершения.

**Входные документы:**

- [Конспект лекции по Инженерии требований](requirements-engineering-summary.pdf) — также доступен в виде глав: [docs/requirements-engineering/](docs/requirements-engineering/00-README.md);
- [Issue: требования, тикеты и Merge Request](REQUIREMENT.md) — копия файла из другого проекта, ради которого проводится данный эксперимент. В документе есть ссылки на документацию, ссылки — не рабочие, текст самодостаточен и без дополнительных файлов.

**Решения и история изменений:** [docs/adr/](docs/adr/README.md) — Architecture Decision Records (ADR). Принцип фиксации — см. раздел [ADR: как фиксируем изменения](#adr-как-фиксируем-изменения).

## ADR: как фиксируем изменения

Когда в ходе эксперимента принимается или меняется решение — в первую очередь о **формулировках инструкций** для ИИ-ассистентов на этапах, но также о процессе и артефактах, — мы фиксируем его в каталоге [docs/adr/](docs/adr/README.md) в форме **ADR** (Architecture Decision Record).

**Зачем.** Текст инструкции (`INSTRUCTIONS.md` в рабочей директории этапа) — источник истины для **исполнения**. ADR — источник истины для **обоснования**: почему инструкция сформулирована именно так, какие альтернативы отвергнуты, что изменилось по итерациям обсуждения с экспериментатором.

**Как.**

1. Каждое принятое решение — отдельный файл: `NNN-краткое-название.md` (например, [`001-analyst-instructions.md`](docs/adr/001-analyst-instructions.md)).
2. Нумерация сквозная по эксперименту: `001`, `002`, `003`…
3. **Принятые ADR не переписываем.** Если решение меняется — добавляем новый ADR со статусом «принято, заменяет ADR-NNN» и описанием delta; старый файл остаётся в истории.
4. В ADR — контекст, решение, обоснование, рассмотренные альтернативы, последствия; при необходимости — хронология итераций до принятия.
5. После принятия ADR обновляем соответствующую инструкцию в рабочей директории этапа.

Если итерация этапа эксперимента неудачна, возвращаемся сюда: правим инструкцию, при существенных решениях — новый ADR.

## Этапы эксперимента

Планируется, что я (пользователь) буду выражать своё мнение только как экспериментатор, то есть только в этом проекте. В ходе выполнения эксперимента я буду подготавливать входные данные, запускать процесс, указывая ИИ-ассистенту на входные данные, следовать указаниям ИИ-ассистента, отвечать на вопросы ИИ-ассистента.

Ты (ИИ-ассистент) подготавливаешь инструкции для ИИ-ассистента каждого этапа и рассказываешь мне, как воспользоваться инструкцией.

Выполнение этапа может быть многоитерационным. Если итерация неудачна — возвращаемся в этот проект, исправляем инструкции и повторяем.

Если этап проводится в несколько шагов, инструкции должны содержать принцип **HITL** для оценки результатов каждого шага.

К проекту подключен MCP-сервер `gitlab-ai-helper` для наблюдения за результатами этапа оформления требований. Дополнительная цель — добавление или модификация инструментария ИИ-агента для удобства использования MCP на последующих этапах. У тебя есть доступ на чтение.
