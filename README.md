# Эксперимент

Был поставлен эксперимент про разработку программного обеспечения. В эксперименте участвовали пользователь и ИИ-ассистент. На каждом этапе выполнялись разные роли.

В данном проекте наша роль была ролью наблюдателей и незримых помощников всего процесса. Мы формулировали и ставили задачи для последующих этапов, анализировали результаты. ИИ-ассистент экспериментатора составлял инструкции для ИИ-ассистентов на каждом этапе. По завершении эксперимента обсудили результаты и сформулировали выводы (ниже).

## Цель эксперимента

Эксперимент реализовал **No- и/или Low-coding подходы в разработке программного обеспечения**. В рамках эксперимента нужно было понять, воспроизвести и описать аспекты **планирования** разработки ПО; на основе планирования — проектирование и разработка.

Параллельная цель (связанная с другим проектом): предложить **простой сценарий обеспечения наличия каждого свойства требования** — см. [REQUIREMENT.md](REQUIREMENT.md).

Параллельная цель (связанная с проектом GitLab AI Helper): по ходу эксперимента накапливать в файле **[FINDINGS.md](FINDINGS.md)** **самодостаточные** рекомендации для ИИ-агента, дорабатывающего тот проект (документ вёлся здесь и переносится в репозиторий GitLab AI Helper). В документе фиксировались обнаруженные проблемы и **желательный вид** решения (без проектирования самого решения). Каждый завершённый этап или существенная итерация могли добавлять новые находки.

Процесс был **многоитерационным**: в любой момент могли поступить новые пожелания; на стадии разработки могла выясниться невозможность или нецелесообразность требований; могла обнаружиться неполнота спецификации — и тогда нужно было возвращаться к общению с заказчиком.

## Роли в процессе


| #   | Роль                    | Назначение                                                                         |
| --- | ----------------------- | ---------------------------------------------------------------------------------- |
| 1   | **Аккаунт-менеджер**    | Сбор информации от заказчика, проведение интервью                                  |
| 2   | **Бизнес-аналитик**     | Анализ собранной информации, подготовка спецификации требований                    |
| 3   | **Продукт-менеджер**    | Формирование задач (Ведущему разработчику и Разработчику), управление приоритетами |
| 4   | **Ведущий разработчик** | Подготовка и изменение технологического стека и окружения разработки               |
| 5   | **Разработчик**         | Создание программного обеспечения                                                  |


На этапе разработки пользователь выступал как **вайб-кодер** — участник без навыков программирования, у которого уже были задачи и приоритеты. Цель предыдущих этапов — обеспечить ему возможность участвовать в разработке ПО. Окружение и стек готовил **Ведущий разработчик** (этап 4), а не Продукт-менеджер.

**Заказчик** — отдельная роль на первом шаге; пользователь играл её при работе с Аккаунт-менеджером. Роль Заказчика отдельно не логировалась — вместо этого фиксировался артефакт работы Аккаунт-менеджера.

ПО для автоматического добавления и связывания требований и задач в GitLab на момент эксперимента **не было готово** — на этапах 2–4 пользователь выполнял эти действия **вручную** (при необходимости с помощью MCP-сервера `gitlab-ai-helper`). Действия пользователя на этапах, где он выполнял роль **Пользователя** (этапы 2–4), а также в роли **Вайб-кодера** (этап 5), **логировались** для анализа эксперимента после его завершения.

**Входные документы:**

- [Конспект лекции по Инженерии требований](requirements-engineering-summary.pdf) — также доступен в виде глав: [docs/requirements-engineering/](docs/requirements-engineering/00-README.md);
- [Issue: требования, тикеты и Merge Request](REQUIREMENT.md) — копия файла из проекта GitLab AI Helper, ради которого проводился данный эксперимент. В документе есть ссылки на документацию, ссылки — не рабочие, текст самодостаточен и без дополнительных файлов.

**Решения и история изменений:** [docs/adr/](docs/adr/README.md) — Architecture Decision Records (ADR). Принцип фиксации — см. раздел [ADR: как фиксировали изменения](#adr-как-фиксировали-изменения).

## ADR: как фиксируем изменения

Когда в ходе эксперимента принимается или меняется решение — в первую очередь о **формулировках инструкций** для ИИ-ассистентов на этапах, но также о процессе и артефактах, — мы фиксируем его в каталоге [docs/adr/](docs/adr/README.md) в форме **ADR** (Architecture Decision Record).

**Зачем.** Текст инструкции (`INSTRUCTIONS.md` в рабочей директории этапа) — источник истины для **исполнения**. ADR — источник истины для **обоснования**: почему инструкция сформулирована именно так, какие альтернативы отвергнуты, что изменилось по итерациям обсуждения с экспериментатором.

**Как.**

1. Каждое принятое решение — отдельный файл: `NNN-краткое-название.md` (например, `[001-analyst-instructions.md](docs/adr/001-analyst-instructions.md)`).
2. Нумерация сквозная по эксперименту: `001`, `002`, `003`…
3. **Принятые ADR не переписываем.** Если решение меняется — добавляем новый ADR со статусом «принято, заменяет ADR-NNN» и описанием delta; старый файл остаётся в истории.
4. В ADR — контекст, решение, обоснование, рассмотренные альтернативы, последствия; при необходимости — хронология итераций до принятия.
5. После принятия ADR обновляем соответствующую инструкцию в рабочей директории этапа.

Если итерация этапа эксперимента неудачна, возвращаемся сюда: правим инструкцию, при существенных решениях — новый ADR.

## Этапы эксперимента

Пользователь выражал своё мнение только как экспериментатор, то есть только в этом проекте. В ходе выполнения эксперимента он подготавливал входные данные, запускал процесс, указывая ИИ-ассистенту на входные данные, следовал указаниям ИИ-ассистента, отвечал на вопросы ИИ-ассистента.

ИИ-ассистент экспериментатора подготавливал инструкции для ИИ-ассистента каждого этапа и рассказывал, как воспользоваться инструкцией.

Выполнение этапа могло быть многоитерационным. Если итерация оказывалась неудачной — возвращались в этот проект, исправляли инструкции и повторяли.

Если этап проводился в несколько шагов, инструкции должны были содержать принцип **HITL** для оценки результатов каждого шага.

К проекту был подключен MCP-сервер `gitlab-ai-helper` для наблюдения за результатами каждого этапа.

## Анализ результатов

### Этап 2 (бизнес-аналитик)

Спецификация требований в целом удовлетворяет критериям IEEE 830 на уровне документа; при переносе в GitLab Issue обнаружен **инфраструктурный разрыв**: система хранения не поддерживает вложения, а модифицируемость совокупности требований не достигается «сама по себе» через проверки отдельной Issue. Подробные находки — в [FINDINGS.md](FINDINGS.md) (самодостаточный документ для переноса в GitLab AI Helper).

После [ADR-019](docs/adr/019-ba-proactive-requirement-quality.md) совокупность формулировок доработана; четыре требования упирались в промпт верификатора, а не только в текст BA. Правка промпта Issue в Helper + повторный прогон: **все** требования с положительным вердиктом ([ADR-020](docs/adr/020-full-issue-verification-pass.md), **F-010**).

### Этап 3 (продукт-менеджер)

Задачи совокупности после той же правки промпта Issue также получили **положительный** вердикт (ADR-020). Поиск узких мест в промпте верификации Merge Request был выполнен, но вывод тогда **не принят** (ADR-020). По итогам этапа 5 система верификации MR признана рабочей ([ADR-027](docs/adr/027-model-choices-and-mr-verification.md)).

### Этапы 4–5 (волна 1 и далее)

Первая волна задач Разработчика закрыта. Задержка: слабая сверка каркаса Ведущего с Issue (E2E) — процесс Ведущего дополнен шагом «добавить/убрать» с ИИ ([ADR-024](docs/adr/024-wave1-lead-scaffold-task-mcp.md)). В `/task`: только MCP для Issue, MR создаёт Пользователь. Для Helper — [F-011](FINDINGS.md): не догонять `glab`, оставить уникальное (история Issue), Wiki — в перспективе.

### Итог продукта и требования

Продукт (сетевой калькулятор) **полностью реализован** по совокупности задач. Low-/No-coding конвейер дошёл до конца: от пожеланий Заказчика через роли до рабочего ПО.

Как в [лекции по инженерии требований](docs/requirements-engineering/01-vvedenie-i-problemy.md) (Брукс): ни одна задача не приносит такого же вреда конечной системе в случае ошибки, как неверное или слабое определение того, что требуется создать. Оформление калькулятора было описано слабо; визуальные эталоны и правила layout оставили простор для спорных, но формально исполнимых постановок.

**Кейс [#51](https://gitlab.apnethq.com/amindubaev-projects/web-calculator/-/issues/51)** (TASK-021: размер виджета и центрирование при смене viewport). При выполнении ИИ-ассистент Разработчика задал вопросы, на которые у вайб-кодера не было ответов → обращение к Продукт-менеджеру → к Бизнес-аналитику: формулировка оказалась **намеренной**. Это был не чистый «конвейер исполнения», а петля уточнения смысла. Менять задачу через Аккаунт-менеджера было бессмысленно: Заказчик уже получил желаемое в требованиях. Важнее вывод: исполнение «как написано» после сверки с BA — не сбой процесса; риск — отсутствие **продуктового вето** (вайб-кодер / PM / будущий тестировщик), который мог бы зафиксировать «сделано по требованию» и отдельно — «требование спорно для UX», без переписывания Issue через AM.

ИИ-ассистент хорошо исполняет постановку; критический анализ смысла остаётся на человеке в роли Пользователя (и на более ранних ролях).

### Этап 5 — наблюдения

1. **Холодный старт волны.** В начале волны задач ИИ-ассистенту Разработчика сложнее всего: мало локального контекста репозитория и конвенций, приходится много помогать. Позже та же сложность задач даётся легче за счёт накопленного контекста проекта, а не «взросления» модели.
2. **Циклы на тестах.** ИИ-ассистент может зациклиться на починке красных тестов и всё равно не решить задачу. На практике вайб-кодер останавливал цикл, просил объяснить проблему и вместе искал причину; иногда решения человека ИИ записывал как «техдолг» без явной договорённости.
3. **Пробелы в задачах при зелёном верификаторе.** ИИ часто не хватало деталей в тексте задачи — вопросы были простые; к Продукт-менеджеру за разъяснениями обращались лишь несколько раз (в т.ч. по #51 — с выходом на BA). Верификатор Issue и исполнитель смотрят на разное: положительный вердикт Issue ≠ исполнимость без уточнений. Большинство пробелов закрывал вайб-кодер из здравого смысла и соседних артефактов — это и плюс (процесс живёт), и риск (молчаливые допущения не попадают в требования).
4. **Недочёты PM не передавались.** Шаг «видимые недочёты → задача PM в текущей волне» на практике не выполнялся. К нему планируется вернуться после реализации семантического поиска требований и задач.
5. **Модели.** `gpt-5-mini` заметно сильнее `gpt-5-nano` — оставляем `gpt-5-mini` основной моделью в GitLab AI Helper. Для разработки Grok 4.5 справляется лучше Composer 2.5; после вердикта системы верификации MR Composer исправляет недочёты с первого раза — **система верификации Merge Request работает** ([ADR-027](docs/adr/027-model-choices-and-mr-verification.md)).

В процессе эксперимента **роли Тестировщика не было**: регрессии и «зелёные, но неверно» ловились опосредованно (вайб-кодер, Code Review Ведущего, точечная приёмка PM).

### Планы на будущее

Инструкции исполнителей эксперимента по итогам **решили не менять**. Ниже — направления, намеченные для следующих итераций процесса / продукта Helper и сжатия ролей.

1. **Прогревочная задача Ведущего разработчика.** Перед боевой волной задач Разработчика — отдельная прогревочная задача, которую Ведущий выполняет **вместе со своим ИИ-ассистентом**: каркас, соглашения, QA/E2E, типичный цикл MR. Цель — не новый продукт, а живой контекст для следующей волны. Критерий успеха: на первой боевой задаче Разработчик задаёт меньше «онбординговых» вопросов.
2. **Бюджет попыток на починку тестов и разбор причины.** Ограничить число циклов «починить тесты → снова красные»; при исчерпании — стоп → объяснение гипотезы (код / тест / требование / окружение) → решение человека о направлении. Явно различать осознанный техдолг (с approve Ведущего) и запрет прятать обход под ярлыком «техдолг» без договорённости.
3. **Видимые недочёты → PM.** Вернуться к передаче видимых недочётов Продукт-менеджеру (новая задача в волне) после реализации семантического поиска требований и задач.
4. **Смещение и слияние ролей; место Тестировщика.** Цепочка из многих ролей нужна, чтобы понять функции; в зрелом low-code сценарии функции должны **перетекать**, столько людей не требуется:

  | В эксперименте было размазано                             | Куда может перетечь                                                     |
  | --------------------------------------------------------- | ----------------------------------------------------------------------- |
  | Уточнение деталей исполнения (не смена смысла требования) | Разработчик (+ локальные ADR), без BA                                   |
  | «Спорный UX / нелогичное, но так написано»                | короткий продуктовый вето (PM или вайб-кодер), без AM                   |
  | Ловля регрессий и «зелёные, но неверно»                   | роль **Тестировщика** (человек+ИИ) или часть функций Ведущего на review |
  | Онбординг в стек/каркас                                   | прогрев Ведущего (п.1), не BA/PM                                        |

   Цель следующих итераций — зафиксировать **обязательные функции**, а не вечную пятёрку ролей. Тестировщик — явный пробел проведённого эксперимента: верификация Issue ≠ приёмка поведения продукта.

Находки для GitLab AI Helper по-прежнему накапливаются в [FINDINGS.md](FINDINGS.md) отдельными записями, когда решено фиксировать разрыв для Helper (не дублируя сюда проектирование решений).
