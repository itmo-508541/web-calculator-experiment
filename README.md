# Эксперимент

Хочу поставить эксперимент про разработку программного обеспечения. В эксперименте будем участвовать мы с тобой (то есть пользователь и ИИ-ассистент). На каждом этапе мы будем выполнять разные роли.

В данном проекте наша роль - наблюдателей и незримых помощников всего процесса. Мы будем формулировать и ставить задачи для последующих этапов. Мы - те, кто ставит эксперимент и анализирует результаты процессов, проходящих на тех этапах. Также ты будешь составлять инструкции для ИИ-ассистента всех этапов. В конце эксперимента мы обсудим результаты эксперимента и ты поможешь сформулировать выводы.

**Входные документы:**

- [Конспект лекции по Инженерии требований](requirements-engineering-summary.pdf) — также доступен в виде глав: [docs/requirements-engineering/](docs/requirements-engineering/00-README.md);
- [Issue: требования, тикеты и Merge Request](REQUIREMENT.md) — копия файла из другого проекта, ради которого проводится данных эксперимент. В документе есть ссылки на документацию, ссылки - не рабочие, текст самодостаточен и без дополнительных файлов.

**Решения и история изменений:** [docs/adr/](docs/adr/README.md) — Architecture Decision Records (ADR) по подготовке инструкций и другим решениям эксперимента. Принцип фиксации — см. раздел [ADR: как фиксируем изменения](#adr-как-фиксируем-изменения).

**Цель эксперимента:** предложить **простой сценарий обеспечения наличия каждого свойства требования**. Реализация сценария не обязательно должна быть автоматической. Цель эксперимента совпадает с целью другого проекта, см. в [REQUIREMENT.md](REQUIREMENT.md).

## ADR: как фиксируем изменения

Когда в ходе эксперимента принимается или меняется решение — в первую очередь о **формулировках инструкций** для ИИ-ассистентов на этапах, но также о процессе и артефактах, — мы фиксируем его в каталоге [docs/adr/](docs/adr/README.md) в форме **ADR** (Architecture Decision Record).

**Зачем.** Текст инструкции (`INSTRUCTIONS.md` в рабочей директории этапа) — источник истины для **исполнения**. ADR — источник истины для **обоснования**: почему инструкция сформулирована именно так, какие альтернативы отвергнуты, что изменилось по итерациям обсуждения с экспериментатором.

**Как.**

1. Каждое принятое решение — отдельный файл: `NNN-краткое-название.md` (например, [`001-analyst-instructions.md`](docs/adr/001-analyst-instructions.md)).
2. Нумерация сквозная по эксперименту: `001`, `002`, `003`…
3. **Принятые ADR не переписываем.** Если решение меняется — добавляем новый ADR со статусом «принято, заменяет ADR-NNN» и описанием delta; старый файл остаётся в истории.
4. В ADR — контекст, решение, обоснование, рассмотренные альтернативы, последствия; при необходимости — хронология итераций до принятия.
5. После принятия ADR обновляем соответствующую инструкцию в рабочей директории этапа.

Если итерация этапа эксперимента неудачна, возвращаемся сюда: правим инструкцию, при существенных решениях — новый ADR.

## Этапы эксперимента

Планируется, что я (пользователь) буду выражать своё мнение только как экспериментатор, то есть только в этом проекте. В ходе выполнения эксперимента я буду лишь подготавливать входные данные, запускать процесс указывая ИИ-ассистенту на входные данные, следовать указаниям ИИ-ассистента, отвечать на вопросы ИИ-ассистента.

Планируется, что ты (ИИ-ассистент) будешь подгодавливать инструкции для ИИ-ассистента каждого последующего этапа, и будешь рассказывать мне (пользователю) как воспользоваться той инструкцией.

Выполнение этапа эксперимента может быть многоитерационным. Если итерация будет признана неуспешной, мы вернёмся в этот проект, чтобы исправить инструкции для ИИ-ассистента, чтобы попробовать выполнить неудавшийся этап ещё раз.

Если этап должен быть проведён в несколько шагов, то инструкции ИИ-ассистенту должны содержать принцип HITL для оценки результатов выполнения шага.

К проекту подключен MCP-сервер `gitlab-ai-helper` для наблюдения за результатами этапа "Оформление требований". Ещё одной дополнительной целью проекта является добавление или модификация набора инструментария ИИ-агента, использующего данный MCP-сервер для удобства его использования на последующих этапах. У тебя есть доступ на чтение.
