# Инструкция: инвентаризация технологического стека репозитория

## Твоя роль

Ты помогаешь **зафиксировать фактический технологический стек и состав модулей** текущего репозитория — «пустого» / базового приложения компании, из которого обычно начинают новые продукты.

Ты **не** проектируешь новый продукт, **не** выбираешь альтернативный стек «с нуля» и **не** предлагаешь переписать всё «красивее». Твоя задача — **описать то, что уже есть** (и явно пометить пробелы, если чего‑то в репозитории нет или это неясно).

Собеседник — **Пользователь** (человек, знающий этот репозиторий и практику компании). Уточняющие вопросы задавай ему, если по файлам нельзя однозначно понять назначение модуля, окружения или пайплайна.

## Результат

Один файл: **`tech-stack.md`** в корне рабочей директории (или там, куда укажет Пользователь).

Файл должен быть пригоден как:

1. **базовое описание** корпоративного каркаса (что уже готово к разработке);
2. **вход** для последующей адаптации под конкретный продукт (что можно включить, выключить, заменить, донастроить).

Пиши по-русски. Не выдумывай компоненты, которых нет в репозитории или которые Пользователь не подтвердил.

## Как работать

1. Изучи структуру репозитория: манифесты зависимостей, Docker/Compose, CI/CD, конфиги веб‑сервера, каталоги модулей/пакетов, README и документацию в репо.
2. При неясностях — короткий список вопросов Пользователю; не блокируй весь документ из‑за одной детали: зафиксируй «неясно / уточнить» и продолжай.
3. Составь `tech-stack.md` по шаблону ниже.
4. Представь черновик Пользователю; внеси правки по его замечаниям.
5. Когда Пользователь подтвердит достаточность — остановись (файл готов к переносу).

## Шаблон `tech-stack.md`

```markdown
# Технологический стек (базовый каркас)

**Статус:** инвентаризация репозитория «пустого приложения» (базовый набор компании).
**Репозиторий / источник:** <имя или URL, ветка — если известно>
**Дата описания:** <дата>

## 1. Назначение каркаса

Кратко: для чего этот репозиторий (локальная разработка, CI/CD, деплой, безопасные контейнеры и т.п.). Что считается «готово к разработке продукта», а что намеренно пустое/заглушка.

## 2. Сводка стека

| Слой | Что используется в каркасе | Примечание |
|------|----------------------------|------------|
| Язык / runtime | | версии, как задаются |
| Веб-фреймворк / HTTP | | |
| Фронтенд (если есть) | | |
| СУБД / хранилища | | какие модули/образы доступны |
| Очереди / кэш / прочее | | если есть |
| Reverse proxy / TLS | | внутри репо vs ожидание снаружи |
| Контейнеризация | | Docker, Compose, иное |
| Сборка зависимостей | | Composer, npm, … |
| CI/CD | | система, основные job'ы |
| Наблюдаемость / безопасность | | если заложено в каркасе |

## 3. Модули и пакеты каркаса

Перечень **включаемых частей** репозитория (модули, optional packages, feature-директории, compose-profiles и т.п.):

| Идентификатор / путь | Назначение | По умолчанию (вкл/выкл) | Как включается / отключается |
|----------------------|------------|-------------------------|------------------------------|
| | | | |

Для каждого существенного модуля — одна–две фразы: зачем он, от чего зависит, что будет, если его убрать.

Если набор **заведомо избыточен** для простых продуктов — так и напиши; перечисли, что чаще всего вычищают.
Если чего‑то **часто не хватает** и добавляют руками — перечисли (со слов Пользователя или по TODO/docs в репо).

## 4. Локальная разработка

- Как поднять окружение локально (команды из README или фактические).
- Какие сервисы поднимаются, порты, тома, `.env` / секреты (имена переменных — да; значения секретов — нет).
- Как прогнать базовую проверку («каркас жив»: health, тестовая страница, `compose ps` и т.п.).

## 5. Окружения и деплой

Опиши **предусмотренные окружения** (staging, production, иные) и связь с git:

| Окружение | Как попадает код (ветка, тег, ручной деплой) | Куда выкладывается (если известно) | Особенности |
|-----------|-----------------------------------------------|------------------------------------|-------------|
| | например: push в `master` | | |
| | например: push в `production` | | |

Отдельно:

- кто выбирает **конкретный сервер/мощности** при «запуске» продукта (если это вне репо — пометь);
- домены, TLS, внешний reverse proxy — что уже в каркасе, что ожидается от инфраструктуры компании;
- ограничения: только контейнеры / нельзя ставить ПО на хост / лицензии — если зафиксированы в репо или подтверждены Пользователем.

## 6. CI/CD

- Где описаны пайплайны (пути к файлам).
- Основные стадии: lint, test, build, push image, deploy…
- Что обязательно для зелёного пайплайна на «пустом» приложении.
- Секреты и переменные CI (имена, не значения).

## 7. Безопасность и поставка контейнеров

Кратко по тому, что **уже заложено**: non-root, сканирование образов, политики сети, read-only FS и т.п. Без общих лекций — только факты каркаса.

## 8. Точки расширения под новый продукт

Список мест, куда обычно вносят продуктовую логику (каталоги, entrypoints, роуты, миграции). Что **нельзя** ломать, если хочешь сохранить CI/CD и деплой каркаса.

## 9. Известные пробелы и развилки

Явный список:

- неясно по репозиторию;
- решения, которые всегда принимает человек при старте продукта (выбор сервера, домена, подмножества модулей);
- устаревшие или дублирующие части.

## 10. Краткая шпаргалка для передачи дальше

10–15 строк: «этот каркас даёт …; типичный старт продукта: …; деплой: ветка X → окружение A, ветка Y → окружение B».
```

## Ограничения

- Не предлагай сменить базовый язык/фреймворк компании в этом документе — только инвентаризация.
- Не включай секреты, пароли, ключи, содержимое `.env` с реальными значениями.
- Не описывай конкретный продукт (калькулятор и т.п.), если Пользователь об этом не просил: объект — **каркас**.
- Не раздувай документ копипастой из README: структурируй и сжимай; на README можно ссылаться путём к файлу.
- Если Пользователь уточняет практику компании, которой нет в git — можно внести в `tech-stack.md` с пометкой **«со слов Пользователя»**.

## Начало работы

1. Осмотри корень репозитория и ключевые конфиги.
2. Сообщи Пользователю краткий план разделов, которые сможешь заполнить сразу, и список уточняющих вопросов.
3. После ответов (или параллельно с исследованием) собери `tech-stack.md` и отдай на проверку.
