# ADR-019: BA — превентивное качество формулировок требований

## Статус

Принято; реализует смену линии [ADR-011](011-ba-proactive-quality-after-verification-run.md); уточняет [ADR-010](010-ba-verification-reaction.md) (реактивный шаг 4 сохраняется, но при правке после вердикта снова опирается на те же четыре свойства). Правки [`business-analyst/INSTRUCTIONS.md`](/home/covex/Calculator/business-analyst/INSTRUCTIONS.md) и [README.md](../../README.md) внесены (2026-07-18).

## Дата

2026-07-18

## Контекст

По [ADR-011](011-ba-proactive-quality-after-verification-run.md) после полного прогона верификации (17/29 готовы, 0 с первого раза) зафиксирована смена линии: BA должен изначально формировать требования со свойствами качества, а не рассчитывать только на шаг 4. Инструкция до этой итерации ещё не менялась.

На практике слабые формулировки `REQ-…` мешают не только верификации самих требований, но и верификации задач PM, опирающихся на эти Issue (пример разбора: #17 / REQ-016 — суть «открытого доступа» читаема, но однозначность, полнота относительно заявленных тезисов и проверяемость частичны).

Согласовано с экспериментатором:

- цель — **достаточно годный первичный** текст Issue (желательно), без жёсткого стопа при большом корпусе и без гарантии «с первого раза»;
- те же четыре свойства напоминать при реакции на отрицательный вердикт (шаг 4);
- не раздувать `REQ-…` реализацией «под верификатор».

Аналог для задач PM — [ADR-017](017-pm-proactive-task-quality.md).

## Решение

### 1. Превентивная проверка на шаге 3

Перед фиксацией текста в Приложении B BA **желательно** для каждого `REQ-…` проходит внутреннюю проверку:

1. **Однозначность**
2. **Полнота** (в границах одной возможности / одного `REQ-…`; критерий проверки покрывает заявленные тезисы)
3. **Непротиворечивость**
4. **Проверяемость**

Отдельный артефакт «отчёт о проходе» не вводится. Гарантия прохождения верификации не требуется.

### 2. Шаг 4 — те же свойства при переформулировании

При переданном отрицательном (или смешанном) вердикте:

- в обсуждении сверять замечания с четырьмя свойствами (отделяя ложные срабатывания и выход за границы `REQ-…`);
- после явного указания Пользователя переформулировать так, чтобы требование снова ими обладало.

HITL и «правка только по явному указанию» из ADR-010 сохраняются.

### 3. Границы превенции

- Не закрывать пробелы догадкой о воле заказчика — запрос к Аккаунт-менеджеру.
- Не добавлять стек, эндпойнты, формат API «лишь бы пройти проверку».
- Восемь характеристик IEEE 830 в начале инструкции остаются рамкой; четыре свойства — операциональный акцент для текста Issue и для петли после верификации.

## Обоснование

- Реализует отложенную правку ADR-011 без смешения «зафиксировали итог прогона» и «внедрили процесс» в одной дате.
- Мягкая формулировка («желательно») снижает риск деградации при большом числе требований, сохраняя целевой ориентир.
- Явное повторение четырёх свойств на шаге 4 связывает превенцию и реакцию одной рамкой (как у PM в ADR-017).

## Рассмотренные альтернативы

| Альтернатива | Почему отклонена |
| --- | --- |
| Только обновить шаг 4 | Не улучшает первичный текст до переноса |
| Жёсткий обязательный гейт «не представляй без прохода» как у PM | Экспериментатор хочет желательность, не стоп при большом корпусе |
| Полный чеклист всех восьми IEEE 830 на шаге 3 | Для операционализации Issue достаточно четырёх свойств верификатора; остальные уже в начале инструкции |
| Суб-агенты как обязательный механизм | ADR-011 допускал; конкретный дизайн не требуется для этой итерации |

## Последствия

- Исполняемая истина для BA — обновлённый `INSTRUCTIONS.md` и настоящий ADR; ADR-011 в части «инструкция пока не меняется» закрыт внедрением.
- Установка ADR-008/010 «не ужесточать первый черновик под верификатор» для **BA** в части четырёх свойств качества **заменена** превентивной желательной проверкой; запрет раздувать реализацией сохраняется.
- README этапа 2 отражает внедрение превенции (ссылка на ADR-019 / ADR-011).
