# ADR-027: Выбор моделей и работоспособность верификации Merge Request

## Статус

Принято; дополняет итог эксперимента (этап 5). В части практики верификации MR снимает отложенность из [ADR-020](020-full-issue-verification-pass.md) §4: система верификации MR **признана рабочей**. Инструкции ролей эксперимента **этим ADR не меняются**. README обновлён (2026-07-21).

## Дата

2026-07-21

## Контекст

После закрытия волн задач Разработчика и реализации продукта (калькулятор) экспериментатор зафиксировал наблюдения о моделях в двух контурах:

1. **GitLab AI Helper** (верификация Issue / MR) — сравнение `gpt-5-mini` и `gpt-5-nano`.
2. **Разработка в Cursor** (ИИ-ассистент Разработчика) — сравнение Grok 4.5 и Composer 2.5; поведение после вердикта бота верификации Merge Request.

Ранее (ADR-020) глубокий разбор и правка промпта верификации MR были **отложены** (вывод анализа тогда не принят). На этапе 5 накопилась практика использования вердиктов MR-верификатора в цикле правки кода.

Отдельно: шаг процесса «видимые недочёты при приёмке → новая задача PM в текущей волне» ([ADR-021](021-stage5-dev-mr-review-pm-handoff.md)) на практике **не выполнялся**; возврат к нему отложен до семантического поиска требований и задач (фиксация в README, не в инструкциях).

## Решение

### 1. Основная модель GitLab AI Helper

- Оставить **`gpt-5-mini`** основной моделью в GitLab AI Helper.
- **`gpt-5-nano`** считать заметно слабее для задач Helper; не использовать как основную.

### 2. Модели разработки

- **Grok 4.5** предпочтительнее **Composer 2.5** для основной разработки (постановка задачи → реализация).
- **Composer 2.5** полезен на шаге после вердикта системы верификации MR: недочёты из вердикта исправляет **с первого раза**.

### 3. Верификация Merge Request

- Система верификации Merge Request в GitLab AI Helper **работает** и даёт полезный сигнал для цикла правки (не только Issue).
- Отложенный в ADR-020 глубокий рефакторинг промпта MR этим ADR **не открывается** заново: достаточно признать работоспособность по итогам практики этапа 5. Дальнейшие правки промпта — по отдельному решению.

### 4. Недочёты → PM

- Не менять инструкции сейчас.
- В README зафиксировать факт невыполнения шага и план вернуться после семантического поиска требований и задач.

## Обоснование

- Разница `gpt-5-mini` vs `gpt-5-nano` существенна для качества верификации; основная модель Helper должна быть сильнее.
- Разные модели лучше на разных фазах: Grok 4.5 — разработка «с нуля по задаче»; Composer 2.5 — точечная реакция на структурированный вердикт верификатора.
- Практический цикл «вердикт MR → правка с первого раза» важнее отложенного теоретического разбора промпта (ADR-020 §4).

## Рассмотренные альтернативы

| Альтернатива | Почему отклонена |
| --- | --- |
| Оставить `gpt-5-nano` основной в Helper | Заметно слабее `gpt-5-mini` |
| Composer 2.5 как основная модель разработки | Хуже Grok 4.5 на основной разработке |
| Снова открыть глубокий рефакторинг промпта MR | Практика уже показала работоспособность; отдельное решение при необходимости |
| Менять инструкции этапа 5 под недочёты→PM сейчас | Экспериментатор: инструкции не менять; вернуться после семантического поиска |

## Последствия

- Для доработки GitLab AI Helper: основная модель — `gpt-5-mini` ([FINDINGS.md](../../FINDINGS.md) при переносе находок не обязан дублировать этот ADR целиком).
- В Cursor-разработке продукта: ориентир — Grok 4.5 на задачу; Composer 2.5 — после вердикта MR-верификатора.
- ADR-020 §4 (отложенность) в части «система не проверена / не используется» замещён настоящим ADR; история отложения разбора промпта сохраняется в ADR-020.
