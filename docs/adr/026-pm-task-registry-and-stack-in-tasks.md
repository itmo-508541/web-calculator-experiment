# ADR-026: PM — реестр задач, тексты в `tasks/`, стек только для рекомендаций модулей

## Статус

Принято; заменяет артефакт `developer-tasks.md` из [ADR-006](006-product-manager-instructions.md) / [ADR-013](013-pm-storage-refs.md); уточняет использование `docs/tech-stack.md` в задачах Разработчику ([ADR-014](014-lead-dev-module-recommendations.md), [ADR-015](015-lead-dev-tech-stack-handoff.md)). Обновлены [`product-manager/INSTRUCTIONS.md`](/home/covex/Calculator/product-manager/INSTRUCTIONS.md), [`tasks-list.md`](/home/covex/Calculator/product-manager/tasks-list.md), каталог [`tasks/`](/home/covex/Calculator/product-manager/tasks/), [README.md](../../README.md) — 2026-07-20. Файл `developer-tasks.md` удалён.

## Дата

2026-07-20

## Контекст

1. Инструкция требовала включать «обязательный контекст» низкоуровневого стека в задачи Разработчику. На практике в Issue появлялись секции «Контекст стека» и критерии вроде `A-*-STACK-01` («стек не подменён»), которые не являются проверяемой приёмкой прикладной задачи до/вне деплоя; контроль стека и так делает Ведущий на Code Review.
2. `developer-tasks.md` разросся (>1000 строк) при параллельном ведении таблицы `tasks-list.md` — дублирование TBD и полный текст в одном файле неудобны.

Согласовано с экспериментатором: не запрещать стек явно списками «нельзя»; убрать поручение вшивать стек — оставить использование стека **для рекомендаций модулей**. Слить учёт в один **реестр** `tasks-list.md`; полные тексты — в `tasks/TASK-NNN.md`; от `developer-tasks.md` отказаться сразу.

## Решение

### 1. Стек в задачах Разработчику

В инструкции: **используй технологический стек для рекомендации модулей** (релевантные пункты из раздела рекомендаций Ведущего — ориентир, не критерий приёмки; пакеты не выдумывать). Отдельного требования копировать низкоуровневый стек / «обязательный контекст стека» в каждую задачу **нет**. Явных запретов критериев `A-*-STACK-*` не вводим: без поручения вшивать стек такие критерии не порождаются. Целостность стека при MR — зона Ведущего (Code Review).

### 2. Артефакты постановки

| Артефакт | Назначение |
| --- | --- |
| `tasks-list.md` | **Реестр задач** — оглавление сформулированных задач: id, заголовок, адресат, `#N` или `TBD`, статус, волна, путь к тексту. Без полных формулировок Issue. |
| `tasks/TASK-NNN.md` | Полный текст одной задачи (для переноса в Issue и правок). |
| `developer-tasks.md` | **Не используется**, не создаётся. |

При новой задаче PM сразу обновляет реестр и файл в `tasks/` (один поток, без двух списков TBD).

### 3. Миграция корпуса

Тексты TASK-001…012 перенесены из бывшего `developer-tasks.md` в `tasks/`; `tasks-list.md` переписан как реестр.

## Обоснование

- Рекомендации модулей — единственная продуктовая нужда «прокинуть стек в задачу»; остальное — шум для верификатора и ложные критерии готовности.
- Один реестр проще для ИИ-ассистента, чем таблица + монолит текстов.
- Файлы по одной задаче удобнее править и диффить по волнам.

## Рассмотренные альтернативы

| Альтернатива | Почему отклонена |
| --- | --- |
| Явный запрет `A-*-STACK-*` и секций стека | Достаточно убрать причину; запреты раздувают инструкцию |
| Постепенный отказ от `developer-tasks.md` | Экспериментатор: слить сейчас |
| Оставить таблицу в `tasks-list.md` плюс `tasks/` | Два формата учёта хуже одного реестра |
