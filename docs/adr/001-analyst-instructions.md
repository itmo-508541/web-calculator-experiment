# ADR-001: Инструкция Аналитика (`analyst/INSTRUCTIONS.md`), версия 1

## Статус

Принято

## Дата

2026-07-05

## Контекст

Эксперимент по разработке ПО проводится в несколько последовательных работ участников. Проект `experimentator` — наблюдатель и постановщик: он готовит инструкции для ИИ-ассистентов на каждом шаге, а экспериментатор анализирует результаты и при неудаче корректирует инструкции.

Первый шаг эксперимента — **сбор требований к ПО на основе документа от заказчика** (см. [README.md](../../README.md)). Рабочая директория участника — `/home/covex/Calculator/analyst`. ИИ-ассистент выступает **Аналитиком**, пользователь — **Заказчиком**.

Цель эксперимента в целом — по итогам пути «документ заказчика → реализация» предложить простые сценарии обеспечения свойств качества требований (см. [REQUIREMENT.md](../../REQUIREMENT.md)). На этом шаге особенно задействованы **корректность** и **полнота совокупности** — свойства, требующие внешнего референта (пожелания заказчика) и человеческого процесса.

Входной документ заказчика намеренно минимален: короткий текст и скриншот — модель «Заказчик знает импульс, но не детали». Инструкция не должна раскрывать предметную область: иначе эксперимент теряет смысл (Аналитик должен сам прочитать документ и выявить пробелы).

История обсуждения и правок — в разделе [Хронология решений](#хронология-решений).

## Решение

Подготовить файл [`/home/covex/Calculator/analyst/INSTRUCTIONS.md`](/home/covex/Calculator/analyst/INSTRUCTIONS.md) со следующей **структурой разделов**:

| Раздел | Назначение |
| --- | --- |
| Твоя роль | Роль Аналитика, модель Заказчика, свойства требований IEEE 830 |
| Входные данные | Документ передаёт Заказчик; без имени файла и без содержания |
| Собеседник | Заказчик как собеседник, без мета-ролей и эксперимента |
| Результат работы | Один артефакт `requirements-elicitation.md` с фиксированной структурой |
| Принцип HITL | Три шага с паузой на оценку Заказчиком |
| Шаг 1–3 | Обязанности Аналитика по каждому шагу |
| Ограничения | Границы работы Аналитика |
| Начало работы | Точка входа в процесс |

Cursor rules для директории `analyst/` **не включены** в v1 — планируются отдельно после стабилизации инструкции.

## Обоснование по разделам

### «Твоя роль» и восемь свойств требований

**Почему так.** Аналитик должен ориентироваться на профессиональную практику, а не на ad hoc формулировки. В [REQUIREMENT.md](../../REQUIREMENT.md) свойства качества — центральная рамка всего проекта; на шаге сбора они задают критерии, к которым должны стремиться итоговые требования.

**Почему все восемь с определениями.** Изначально в цели эксперимента фигурировало исключение «Приоритизация» как не автоматизируемой в целевом проекте. Экспериментатор уточнил: свойство нельзя «забывать» — Аналитик должен о нём знать, даже если последующие автоматические сценарии его не проверяют. Поэтому в инструкции перечислены все восемь свойств IEEE 830-1998 с краткими определениями (источник формулировок — конспект лекции, [docs/requirements-engineering/](../../docs/requirements-engineering/00-README.md) и [REQUIREMENT.md](../../REQUIREMENT.md)).

**Почему акцент на «импульсе» Заказчика.** Отражает замысел эксперимента и поговорку «пользователь сам не знает, чего хочет»: детали извлекаются интервью, а не предполагаются в инструкции.

**Почему «без технических деталей реализации».** Граница между требованием и проектным решением — из лекции и [REQUIREMENT.md](../../REQUIREMENT.md); нарушение на этом шаге исказило бы последующие этапы.

### «Входные данные»

**Почему без имени файла.** Документ передаёт Заказчик при старте (вложение или указание в директории). Имя файла — деталь окружения, а не содержание инструкции; фиксация `customer-wishes.pdf` создавала бы лишнюю связь с конкретным артефактом эксперимента.

**Почему «прочитай самостоятельно».** Явный запрет домысливать содержание до чтения — защита от утечки предметной области через инструкцию и через «знание» модели из контекста экспериментатора.

### «Собеседник»

**Почему «собеседник — Заказчик», а не «играет роль».** ИИ-ассистент не знает об эксперименте; для него это обычная задача сбора требований. Пользователь **есть** Заказчик, а не актёр. Формулировки про «роли эксперимента» убраны как лишние и потенциально раскрывающие мета-контекст.

**Почему прерывание работы.** Экспериментатор сохраняет право остановить процесс; для Аналитика это нейтральная инструкция «остановись и жди», без упоминания эксперимента.

### «Результат работы» (ранее «Результат этапа»)

**Почему один файл `requirements-elicitation.md`.** Следующий шаг эксперимента («Оформление требований») потребляет один согласованный артефакт; разбиение на несколько файлов усложнило бы передачу без выигрыша для эксперимента.

**Почему фиксированная структура из четырёх разделов.** Соответствует трём шагам работы плюс итоговый список требований; обеспечивает **отслеживаемость** (связь требования с источником) и **модифицируемость** (раздельные разделы). Фраза «можешь уточнить по ходу» удалена: инструкция должна быть применимой as-is, без делегирования структуры модели.

**Почему не «этап».** Слово «этап» в инструкции относится к шагам **эксперимента** в `experimentator`, а не к работе Аналитика. Для Аналитика — **работа** из трёх **шагов**; переименование снижает путаницу мета-уровней.

### «Принцип HITL» и три шага

**Почему HITL.** Требование [README.md](../../README.md): многошаговые инструкции должны содержать human-in-the-loop. На этом шаге HITL одновременно моделирует **акт согласования с заказчиком** (важно для корректности по [REQUIREMENT.md](../../REQUIREMENT.md)) и даёт экспериментатору точки контроля без раскрытия эксперимента.

**Почему три шага: анализ → план → интервью.** Повторяет классическую схему выявления требований из лекции ([глава 04](../../docs/requirements-engineering/04-vyyavlenie-trebovaniy.md)): сначала понимание артефакта, затем подготовка опросника (метод «Интервью»), затем извлечение информации. Совпадает с формулировкой задачи в README.

**Почему глубина интервью не задана в инструкции.** Экспериментатор указал: охват и детализацию определяет Аналитик по результатам анализа документа; подсказки по темам вопросов (референсы, scope «будущих режимов» и т.п.) искажали бы эксперимент.

**Почему вопросы порциями на шаге 3.** Заказчик-модель не способен ответить на длинный опросник сразу; порционность — практический приём из сценария «скудного заказчика».

**Почему нет критериев успеха в инструкции.** Экспериментатор: заранее заданные критерии упростили бы эксперимент; успех всего эксперимента — факт реализации проекта в конце цепочки.

### «Ограничения»

**Почему запрет GitLab/Issue.** Оформление требований — отдельный шаг эксперимента (`/Calculator/mcp`); смешение нарушило бы разделение ролей и наблюдаемость по шагам.

**Почему без упоминания эксперимента в ограничениях.** Аналитик не должен знать о следующих шагах эксперимента — только о границе своей задачи.

### «Начало работы»

**Почему явная последовательность из трёх пунктов.** Снижает риск, что модель начнёт интервью до анализа документа или пропустит создание файла результата.

## Рассмотренные альтернативы

| Альтернатива | Почему не выбрана |
| --- | --- |
| Несколько выходных файлов (анализ, план, протокол, требования) | Усложняет передачу на следующий шаг; экспериментатор выбрал один md |
| Указание имени `customer-wishes.pdf` и описание предметной области в инструкции | Раскрывает эксперимент, снимает нагрузку с Аналитика по самостоятельному анализу |
| Фиксированный чек-лист вопросов (API, UI, эталон и т.д.) | Подменяет работу Аналитика; вопросы должны «родиться» из документа |
| Критерии успеха и метрики (N требований, нет TBD) | По решению экспериментатора — преждевременно для эксперимента |
| Включение Cursor rules в v1 | Отложено: сначала стабилизировать текст инструкции |
| Исключение «Приоритизация» из списка свойств | Экспериментатор: свойство важно помнить, хотя автоматизация в целевом проекте не планируется |
| Термин «этап» внутри инструкции Аналитика | Путаница с этапами эксперимента; заменено на «работа» / «шаг» |

## Последствия

### Положительные

- Инструкция пригодна для «слепого» запуска: Аналитик получает только инструкцию и документ от Заказчика.
- Артефакт `requirements-elicitation.md` структурирован для следующего шага («Оформление требований»).
- HITL даёт точки исправления итерации без изменения инструкции.
- ADR фиксирует *почему*, не дублируя полный текст инструкции — удобно для ревизий v2, v3.

### Отрицательные / риски

- Без Cursor rules поведение модели может сильнее зависеть от формулировки первого сообщения Заказчика.
- Отсутствие критериев успеха затрудняет формальное «закрытие» шага — оценка субъективна до финала эксперимента.
- Полный текст свойств в инструкции увеличивает объём prompt; при деградации качества возможен перенос определений в приложение со ссылкой.

## Примечание о версионировании

- **ADR-001** — первое принятое решение об инструкции Аналитика.
- При следующих правках `INSTRUCTIONS.md` создавать **новый ADR** (`002-…`, `003-…`), не переписывая ADR-001: в новом документе — delta, статус «заменяет ADR-NNN» при необходимости.
- Текст инструкции — источник истины для исполнения; ADR — источник истины для **решений о формулировках**. Список ADR — [README.md](README.md).
