# ADR-017: PM — превентивное качество формулировок задач

## Статус

Принято; заменяет [ADR-008](008-pm-verification-reaction.md) в части «без ужесточения первого черновика под верификатор»; сохраняет из ADR-008 реактивный разбор верификации только после передачи результата Пользователем и правку только по явному указанию. Уточняет [ADR-006](006-product-manager-instructions.md) и [ADR-016](016-pm-wave-granularity-and-task-links.md). Правки [`product-manager/INSTRUCTIONS.md`](/home/covex/Calculator/product-manager/INSTRUCTIONS.md) и [README.md](../../README.md) внесены (2026-07-16).

Инструкцию Бизнес-аналитика **этим ADR не меняем** (линия для BA — [ADR-011](011-ba-proactive-quality-after-verification-run.md), внедрение — отдельной итерацией).

## Дата

2026-07-16

## Контекст

На практике задачи PM (как и требования BA) редко проходят верификацию Issue с первой попытки. Система верификации нацелена в первую очередь на проверку правок человеком, но ИИ-ассистент может изначально писать текст ближе к критериям верификатора. Гарантия «с первого раза» не нужна; повторный прогон и ложные срабатывания допустимы. Утомительна именно многократная **подгонка** текста после каждого прогона.

Критерии верификатора (однозначность, полнота описания отдельной Issue, непротиворечивость со связанными, проверяемость) и загрузка связанных на глубину не более трёх уровней совпадают с тем, что уже частично требовалось от качественной задачи, но инструкция PM явно запрещала закладывать это превентивно (ADR-008).

Согласовано с экспериментатором: для PM ввести обязательный внутренний проход по этим свойствам **до** HITL и переноса; реактивный шаг после верификации сохранить.

## Решение

### 1. Превентивная проверка качества задачи

Перед представлением задачи / волны Пользователю PM для **каждой** Issue проверяет:

1. **Однозначность** — одна трактовка; без расплывчатых слов без уточнений.
2. **Полнота описания** — контекст, условия/вход, ожидаемый результат, критерии приёмки (полнота всей системы не оценивается).
3. **Непротиворечивость** — внутри текста; относительно требований «выполнить» / «не нарушить»; относительно связанных по ссылкам на глубину ≤ 3; относительно `tech-stack.md`. Явный конфликт — назвать Пользователю, не смягчать.
4. **Проверяемость** — из текста понятно, как убедиться в выполнении.

Отдельный артефакт «отчёт о проходе» не вводится.

### 2. Границы превенции

- Не закрывать пробелы выдуманными продуктовыми или техническими решениями.
- Не добавлять лишние ссылки ради графа для верификатора; анализ связанных на глубину ≤ 3 — да, раздувание ссылок — нет.
- Не раздувать Issue деталями реализации вне требований / стека / договорённости с Пользователем.

### 3. Реактивный шаг сохраняется

ADR-008 в части «разбор только если Пользователь передал результат» и «правка только по явному указанию» остаётся в силе. Превенция не отменяет повторную верификацию.

### 4. Структура текста задачи

В инструкции уточнены элементы содержания (контекст, условия, ожидаемый результат, проверяемые критерии) наряду с прежними адресатом, объёмом, трассировкой и опциональной связью «дополняет».

## Обоснование

- Стоимость реактивной подгонки выше стоимости одного внутреннего прохода при формулировании.
- Критерии верификатора — те же свойства качественной задачи, а не чуждый «формат ради галочки».
- Отказ от гарантии «с первого раза» сохраняет HITL и терпимость к ложным срабатываниям.

## Рассмотренные альтернативы

| Альтернатива | Почему отклонена |
| --- | --- |
| Оставить только реактивный шаг ADR-008 | Утомительная подгонка на практике |
| Требовать гарантию прохождения верификации | Нереалистично; ложные срабатывания |
| Отдельный файл отчёта о превентивной проверке | Лишний артефакт; достаточно проверенного текста |
| Сразу та же правка для BA | Экспериментатор: сначала PM; BA — следующей итерацией |
