# ADR-015: ИИ Ведущего — фиксация стека из репозитория Разработчика

## Статус

Принято; заменяет [ADR-014](014-lead-dev-module-recommendations.md) в части **процесса ИИ-ассистента Ведущего** и **формата / входа `tech-stack.md`**; сохраняет из ADR-014 контракт слоёв (низкоуровневый стек обязателен для Разработчика; рекомендации модулей — ориентир, не приёмка) и обязанность PM прокидывать рекомендации при их наличии. Обновлены [`senior-developer/INSTRUCTIONS.md`](/home/covex/Calculator/senior-developer/INSTRUCTIONS.md) и [README.md](../../README.md) (этап 4).

## Дата

2026-07-15

## Контекст

После ADR-014 инструкция Ведущего опиралась на **базовый** корпоративный `tech-stack.md` (инвентаризация «пустого приложения») и адаптацию «включить / исключить» модули. На практике:

- инвентаризация по `docs/tech-stack-inventory-INSTRUCTIONS.md` получилась избыточной для передачи PM;
- работу в репозитории продукта Ведущий выполняет в **директории Разработчика**; ИИ в каталоге Ведущего помогает зафиксировать итог, а не заново проектировать каркас «по базовому tech-stack.md»;
- инструкция должна быть **проект-агностичной** (без привязки к конкретному продукту вроде калькулятора).

Согласовано с экспериментатором:

1. PM ставит задачу Ведущему (окружение, CI/CD, возможность рекомендаций).
2. Ведущий наполняет репозиторий в директории Разработчика, поднимает staging; production — перед релизом.
3. Ведущий сообщает ИИ номер задачи и путь к директории Разработчика.
4. ИИ читает задачу через MCP и собирает `tech-stack.md` в каталоге Ведущего.
5. Цель ИИ — краткий стек + (опционально) рекомендации после вопросов Ведущему.

## Решение

### 1. Роль ИИ-ассистента Ведущего

| Аспект | Решение |
| --- | --- |
| Каталог ИИ | `/home/covex/Calculator/senior-developer` |
| Работа в коде продукта | Ведущий (Пользователь) в директории Разработчика |
| Вход для ИИ | Номер Issue + путь к директории Разработчика |
| Чтение задачи | MCP (`load-requirement` и аналоги) |
| Выход | `tech-stack.md` **в каталоге Ведущего**; передача PM — руками Ведущего (с возможной ручной правкой) |

Базовый корпоративный `tech-stack.md` эксперимента **не** является входом ИИ-ассистента Ведущего.

### 2. Формат `tech-stack.md`

- Краткий маркированный список атрибутов стека (уровень сводки README продукта), не полная инвентаризация.
- Описывается **точка отсчёта**: только то, что есть сейчас. Формулировки «убрали / исключили относительно каркаса» запрещены.
- Чеклист атрибутов — в инструкции Ведущего (runtime, фреймворк, менеджер зависимостей, хранилища, контейнеры, CI/CD, init/deploy-инструменты, QA, окружения, локальный запуск ссылкой, HTTPS/домен при известности).
- Источник фактов: `README.md` в директории Разработчика → анализ репозитория → вопросы Ведущему при пробелах.

### 3. Рекомендации

- ИИ **обязан спросить** Ведущего о рекомендациях по классам задач из Issue.
- Ведущий может дать рекомендации или не дать — его решение.
- Раздел в файле — **только если есть хотя бы одна** рекомендация; иначе раздел опускается без явной пометки об отказе.
- Смысл рекомендаций как в ADR-014: не требования, не приёмка; PM передаёт ориентиры Разработчику при наличии раздела.

### 4. Сохраняется из ADR-014

| Слой | Обязательность для Разработчика |
| --- | --- |
| Низкоуровневый стек | Обязателен |
| Рекомендации модулей | Не обязательны |

Инструкция PM (прокидывать рекомендации, не назначать модули сам) не меняется этим ADR.

## Обоснование

- Один короткий артефакт достаточен PM; детали запуска живут в README/docs репозитория продукта.
- Точка отсчёта снимает ложную зависимость от файла инвентаризации каркаса и упрощает формулировки.
- Разделение «Ведущий делает репо / ИИ фиксирует стек» соответствует фактическому HITL-процессу.

## Рассмотренные альтернативы

| Альтернатива | Почему отклонена |
| --- | --- |
| Вход = базовый `tech-stack.md` инвентаризации | Избыточен; не нужен при анализе готового репо Разработчика |
| Развёрнутый шаблон инвентаризации как выход для PM | Слишком много лишнего для PM и задач Разработчика |
| Раздел «рекомендаций нет» при отказе Ведущего | Лишний шум; достаточно опустить раздел |
| `tech-stack.md` только в директории Разработчика | Отклонено: файл готовится у Ведущего, он сам передаёт PM |
