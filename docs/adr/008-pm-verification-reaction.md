# ADR-008: Реакция Продукт-менеджера на верификацию Issue

## Статус

Принято, уточняет [ADR-006](006-product-manager-instructions.md) и [ADR-007](007-pm-task-trace-roles.md); правки [`product-manager/INSTRUCTIONS.md`](/home/covex/Calculator/product-manager/INSTRUCTIONS.md) внесены (2026-07-11). Инструкцию Бизнес-аналитика **не** меняем.

## Дата

2026-07-11

## Контекст

На этапе 3 задачи переносятся в GitLab Issue; система верификации (GitLab AI Helper) может выдать замечания по свойствам Issue (однозначность, полнота описания, непротиворечивость, проверяемость) или вердикт «не готово к разработке». Замечаний может не быть.

Прогон показал: часть замечаний спорны или ложны для **задач** Разработчика; часть — полезны. Обсуждение ведёт Пользователь с Продукт-менеджером. Исправлять Issue без согласования нельзя: сначала принципы и детали, затем правка по явному указанию.

Экспериментатор отверг идею заранее ужесточать первую формулировку задач «под верификатор». Инструкцию BA на этом шаге не трогаем: нужный инструментарий (семантический поиск по требованиям) ещё не готов.

## Решение

### 1. Новый подшаг 3.C в инструкции PM

После переноса волны в систему хранения:

- разбор верификации — **только если** Пользователь **передал** её результат; иначе не инициировать;
- сначала обсуждение (включая ложные срабатывания и спорные пункты), без немедленной правки;
- обновление `developer-tasks.md` / Issue — **только после явного указания** Пользователя;
- цель переформулирования — чтобы задача могла быть принята / считалась готовой к разработке;
- цикл повторяется до подтверждения готовности к разработке.

Прежние 3.C / 3.D сдвигаются в 3.D / 3.E. К ожиданию реализации волны — только после готовности к разработке.

### 2. Без ужесточения первого черновика

В инструкцию **не** добавляются правила «сразу писать сверхдетально под верификатор». Достаточность уточняется реактивно, после факта верификации и обсуждения.

### 3. Отдельный файл обсуждения не вводим

Правила живут в `INSTRUCTIONS.md` (рядом с HITL). Отдельный артефакт обсуждения верификации не заводим: обсуждение — в диалоге с Пользователем.

### 4. BA без изменений

Инструкция [`business-analyst/INSTRUCTIONS.md`](/home/covex/Calculator/business-analyst/INSTRUCTIONS.md) на этом решении не меняется.

## Обоснование

- Явное указание на обновление отделяет согласование от мутации Issue и снижает риск правок «по ходу» разговора.
- Триггер «если Пользователь передал» учитывает случай без замечаний и не блокирует процесс ожиданием верификации.
- Реактивная переформулировка соответствует итеративности эксперимента лучше, чем превентивная сверхдетализация.
- Один `INSTRUCTIONS.md` — единая точка входа для агента-исполнителя.

## Рассмотренные альтернативы

| Альтернатива | Почему отклонена |
| --- | --- |
| Отдельный файл «обсуждение верификации» | Дублирует чат; правила процесса должны быть в инструкции |
| Ужесточить первую формулировку задач под верификатор | Явно отвергнуто экспериментатором |
| Та же петля сразу в инструкции BA | ПО/семантический поиск не готовы; преждевременно |
| PM сам правит Issue при любом замечании Helper | Без HITL и явного указания Пользователя |

## Последствия

- После создания Issue волны PM ждёт либо передачи результата верификации, либо подтверждения готовности к разработке без разбора.
- Правки текстов задач после верификации согласованы и трассируемы к указанию Пользователя.
- ADR-006/007 остаются в силе по стеку, волнам и ролям трассировки; настоящий ADR — delta про верификацию.
