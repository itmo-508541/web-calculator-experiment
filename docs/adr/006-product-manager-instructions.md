## Контекст

Этап 2 завершён: есть спецификация требований и таблица соответствий [`requirements-list.md`](/home/covex/Calculator/business-analyst/requirements-list.md) (`REQ-…` ↔ `#<номер>` ↔ заголовок). В README этап 3 был помечен TBD.

Экспериментатор задал содержание инструкции Продукт-менеджера:

1. Передать PM полную спецификацию и таблицу соответствий требований.
2. PM готовит задачи для разработчика в md-файле и **аналогичную** таблицу соответствий задач.
3. В тексте задачи — понятное для ИИ и разработчика описание; обязательная отсылка `REQ-… #<номер>`.
4. **Шаг 1** — выяснение серверных ресурсов у Пользователя; ко **шагу 2** только если информации достаточно.
5. **Шаг 2** — предложение стека с обоснованием, учёт ограничений; к **шагу 3** только после утверждения стека.
6. **Шаг 3** — постановка задач **волнами**: план групп допустим, полный бэклог сразу — вреден; сначала блокирующие задачи → реализация → проверка с Пользователем → следующая волна.

Ранее в [ADR-002](002-roles-and-account-manager.md) этап 3 описывался как «приоритизированный список задач» без явного выбора стека и без запрета на полный бэклог. Настоящий ADR уточняет процесс.

## Решение

### 1. Инструкция Продукт-менеджера

Файл [`product-manager/INSTRUCTIONS.md`](/home/covex/Calculator/product-manager/INSTRUCTIONS.md).

| Аспект | Решение |
| --- | --- |
| Роль | Согласование стека + итеративная постановка задач разработчику |
| Изоляция | ИИ-ассистент **не знает** об эксперименте; в инструкции нет ссылок на `experimentator` |
| Собеседник | **Пользователь** (ресурсы, утверждение стека, HITL, перенос в GitLab, приёмка волн) |
| Вход | Спецификация + `requirements-list.md` (передаёт Пользователь) |
| Артефакты | `tech-stack.md`, `task-plan.md`, `developer-tasks.md`, `tasks-list.md` |
| Трассировка | В тексте задачи: `REQ-001 #25` (ID спецификации + номер в системе хранения) |
| HITL / гейты | Шаг 2 ← достаточность ресурсов; шаг 3 ← утверждённый стек; между волнами — приёмка реализации |

### 2. Три шага с жёсткими условиями перехода

1. **Ресурсы** — вопросы о серверной среде; без достаточной информации стек не предлагать.
2. **Стек** — предложение, обоснование, коррекция по ограничениям; без утверждения задачи не готовить.
3. **Задачи** — план групп + формулирование **только** ближайшей (блокирующей) волны; цикл «постановка → реализация Разработчиком → проверка → следующая волна».

### 3. Обновление README

В разделе этапа 3: ссылка на инструкцию, уточнённые вход/результат/задача, статус «инструкция подготовлена».

## Обоснование

- Выбор стека **до** задач снижает риск переписывания бэклога под несовместимую среду.
- Гейт «достаточно ресурсов» отделяет сбор фактов от проектирования и не даёт PM гадать.
- Волновой бэклог соответствует многоитерационности эксперимента и снижает вред от преждевременной детализации.
- Формат `REQ-… #<номер>` сохраняет отслеживаемость к спецификации и к Issue в системе хранения после ручного переноса требований.

## Рассмотренные альтернативы

| Альтернатива | Почему отклонена |
| --- | --- |
| Сразу полный список задач без стека | Противоречит запросу экспериментатора; высокий риск несовместимости со средой |
| Стек выбирать Разработчику | На этапе 3 нужен согласованный с Пользователем каркас до кодирования; PM отвечает за приоритеты и реализуемость плана |
| Одна задача = одно требование всегда | Иногда нужна инфраструктурная задача, разблокирующая несколько `REQ-…`; трассировка всё равно обязательна |
| PM сам создаёт Issue через MCP | Как на этапе 2: автоматизация не готова / не входит в роль; перенос — Пользователь |

## Последствия

- Каталог `product-manager/` получает рабочую инструкцию; прогон этапа можно начинать.
- Инструкция Разработчика (этап 4) по-прежнему TBD; она должна опираться на утверждённый стек и волны задач.
- При существенном изменении гейтов или артефактов — новый ADR, не правка настоящего «задним числом».
