# ADR-007: Уточнения инструкции Продукт-менеджера

## Статус

Принято, уточняет [ADR-006](006-product-manager-instructions.md); правки [`product-manager/INSTRUCTIONS.md`](/home/covex/Calculator/product-manager/INSTRUCTIONS.md) внесены (2026-07-10).

## Дата

2026-07-10

## Контекст

После подготовки инструкции этапа 3 экспериментатор уточнил три пункта формулировок.

## Решение

### 1. Роль ссылки на требование в тексте задачи

В разделе «Текст задачи: требования к содержанию» у каждой трассировки `REQ-… #<номер>` указывается роль:

- **выполнить** — задача реализует / закрывает требование (или его часть);
- **не нарушить** — задача не обязана закрывать требование, но не должна ему противоречить.

### 2. Термин «Разработчик»

Из инструкции PM исключено понятие «вайб-кодер». Задачи адресованы **Разработчику** и должны быть понятны и человеку, и ИИ-ассистенту.

### 3. «Текущий релиз» вместо v1

Инструкция не привязана к конкретной версии продукта. Вместо «v1» используется формулировка **«текущий релиз»** (применима к любому релизу / версии).

## Обоснование

- Ссылка «не нарушить» нужна, когда задача затрагивает смежные инварианты (изоляция, эталон поведения, уже принятые свойства) без цели закрыть их в этой волне.
- «Вайб-кодер» — термин эксперимента, а не роли в инструкции исполнителя; в инструкции PM его быть не должно.
- Привязка к «v1» сужала инструкцию; процесс постановки один и тот же для любого текущего релиза.

## Последствия

- Тексты задач в `developer-tasks.md` должны явно помечать роль каждой ссылки `REQ-… #<номер>`.
- ADR-006 остаётся в силе по структуре шагов и артефактов; настоящий ADR фиксирует только delta формулировок.
