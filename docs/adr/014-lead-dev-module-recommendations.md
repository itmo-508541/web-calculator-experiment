# ADR-014: Рекомендации модулей Ведущего и жёсткий низкоуровневый стек

## Статус

Принято; уточняет [ADR-009](009-senior-developer-and-pm-stack.md); создан [`senior-developer/INSTRUCTIONS.md`](/home/covex/Calculator/senior-developer/INSTRUCTIONS.md); правки [`product-manager/INSTRUCTIONS.md`](/home/covex/Calculator/product-manager/INSTRUCTIONS.md) и [README.md](../../README.md) внесены (2026-07-15).

**В части процесса ИИ-ассистента Ведущего и формата / входа `tech-stack.md` заменён [ADR-015](015-lead-dev-tech-stack-handoff.md).** Контракт слоёв (низкоуровневый стек обязателен; рекомендации — ориентир) и delta для PM сохраняют силу, пока не отменены отдельно.

## Дата

2026-07-15

## Контекст

После передачи окружения Ведущим Продукт-менеджер и Разработчик по факту заново «выдумывали» средства решения прикладных задач. Нужно сузить inventiveness по модулям **без** превращения рекомендаций в требования и без того, чтобы PM сам назначал инструменты (у PM обычно нет этой компетенции).

Согласовано:

- Ведущий готовит окружение и **включает** потенциально полезные модули / **исключает** ненужные уже на этапе каркаса;
- в `tech-stack.md` — опциональный раздел рекомендаций модулей по классам задач из задачи PM;
- рекомендации не обязательны, не критерии приёмки; помогают поиску аналогов и выбору;
- Разработчик **обязан** сохранять низкоуровневый стек (runtime/версии, CI/CD, менеджер зависимостей вроде Composer) и **не обязан** следовать рекомендациям конкретных модулей в том же стеке;
- задача PM Ведущему — только классы потребностей и ограничения, без длинного списка модулей;
- PM в задачах Разработчику **включает** рекомендации Ведущего (не изобретает инструменты сам).

Риск «каркас с JWT vs открытый API» закрывается адаптацией окружения Ведущим (исключение лишних модулей под ограничения задачи), а не формулировкой must в Issue Разработчика.

## Решение

### 1. Контракт слоёв стека

| Слой | Кто задаёт | Обязательность для Разработчика |
| --- | --- | --- |
| Низкоуровневый стек (runtime и версии, CI/CD, менеджер зависимостей, контракт поставки) | Ведущий (результат в окружении + `tech-stack.md`) | **Обязателен** — не ломать и не подменять |
| Модули/пакеты в рамках того же стека | Ведущий включает в окружение; опционально рекомендует по классам задач | Рекомендации **не обязательны**; выбор модуля — за Разработчиком (ориентир передан через PM) |

### 2. `tech-stack.md`

- Обязательное ядро: фактологическое описание адаптированного стека и окружения.
- Раздел **«Рекомендации модулей по классам задач»** — **опционален** (пустой / частично / по всем классам / отсутствует).
- Рекомендации ≠ требования ≠ приёмка.

### 3. Инструкция Ведущего

Файл [`senior-developer/INSTRUCTIONS.md`](/home/covex/Calculator/senior-developer/INSTRUCTIONS.md): вход = базовый `tech-stack.md` + Issue PM; работа = адаптация окружения + итоговый `tech-stack.md`; HITL с Пользователем-Ведущим.

### 4. Инструкция PM (delta)

- В задаче Ведущему: классы потребностей и ограничения; **не** длинный перечень модулей.
- После получения `tech-stack.md`: в задачах Разработчику кратко передать **релевантные** рекомендации Ведущего (если раздел есть); **не** формулировать самостоятельный выбор инструментов «из головы».
- Рекомендации в тексте задачи Разработчику — ориентир, **не** критерий готовности, если Пользователь явно не решит иначе.

## Обоснование

- Ограничение инструментария на стороне компетенции (Ведущий), как уже сделано для окружения.
- Опциональность рекомендаций не блокирует минимальный каркас и не раздувает верификацию входной задачи Ведущему.
- PM как «канал» рекомендаций отделяет продуктовую постановку от инженерного выбора модулей.

## Рассмотренные альтернативы

| Альтернатива | Почему отклонена |
| --- | --- |
| Рекомендации обязательны / критерий приёмки | Отвергнуто экспериментатором |
| PM сам выбирает модули для Разработчика | Нет компетенции; inventiveness PM |
| Длинный список модулей во входной задаче Ведущему | Дублирует работу Ведущего; вредно для верификации Issue |
| Отдельный файл вместо раздела в `tech-stack.md` | Отвергнуто: один артефакт передачи PM |

## Последствия

- Этап 4 получает исполняемую инструкцию ИИ-ассистента.
- PM обязан читать и прокидывать рекомендации при их наличии; отсутствие раздела — норма.
- Инструкция Разработчика (этап 5) позже должна явно закрепить «низкоуровневый стек обязателен / модульные рекомендации — нет».
