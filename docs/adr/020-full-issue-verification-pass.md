# ADR-020: Полный проход верификации Issue — BA, промпт Helper, задачи; MR отложен

## Статус

Принято; фиксирует исход итерации после [ADR-019](019-ba-proactive-requirement-quality.md). Инструкции ролей **этим ADR не меняются**. FINDINGS дополнен (**F-010**); README этапов 2–3 обновлён (2026-07-18).

## Дата

2026-07-18

## Контекст

После внедрения превентивного качества формулировок требований (ADR-019) экспериментатор с Бизнес-аналитиком прошёл **всю** спецификацию и обновил формулировки `REQ-…` / Issue почти по всему корпусу.

Четыре требования (**REQ-004**, **REQ-016**, **REQ-018**, **REQ-019**) при тогдашнем промпте верификатора Issue оставались с отрицательным вердиктом. Совместный вывод: при корректных (с точки зрения инструкции BA) границах `REQ-…` пройти верификацию **невозможно** — узкие места в промпте GitLab AI Helper, а не только в тексте требований.

Экспериментатор проанализировал промпт верификатора Issue, внёс правки в продукт Helper. После этого **все** требования и **все** задачи корпуса получили **положительный** вердикт системы верификации.

Аналогичный поиск узких мест по промпту верификации **Merge Request** был выполнен, но вывод экспериментатору **не понравился**; дополнительные исследования и правки MR-промпта **отложены**.

## Решение

### 1. Исход по требованиям (этап 2)

- Превентивная линия ADR-019 на практике применена к корпусу: правки формулировок внесены везде, кроме четырёх «непроходимых» при старом промпте.
- После правки промпта верификатора Issue в GitLab AI Helper: **все** `REQ-…` / связанные Issue требований — вердикт «готово» (положительный).
- Ранее зафиксированный прогон ADR-011 (**17/29**, **0** с первого раза) остаётся историческим снимком; актуальный статус корпуса — полный положительный проход после ADR-019 + правки промпта.

### 2. Исход по задачам (этап 3)

- **Все** задачи (Issue задач), прогнанные через верификацию, получили положительный вердикт после той же линии правок промпта Issue (критерии общие для требований и задач).

### 3. Находка для Helper — F-010

Зафиксировать в [FINDINGS.md](../../FINDINGS.md) разрыв промпта верификатора Issue относительно поведенческих требований с трассировкой и осознанными границами (см. **F-010**). Желательный вид решения — способности промпта/оценки, уже частично воплощённые правкой; критерий регрессии: поведенческие `REQ-…` с явным «не задаётся» и ссылками `#N` не должны систематически получать отрицательный вердикт.

Частичное пересечение с **F-009** (обход связанных Issue): в желаемом виде для Issue — сужение фокуса связей; полное снятие противоречия «трассировка ↔ верификация» по-прежнему может потребовать отдельных решений продукта.

### 4. Merge Request — отложено

Поиск узких мест в промпте верификации MR **не принимается** как основание для правок в этой итерации. Повторный анализ и возможные изменения — отдельной итерацией эксперимента / Helper; в FINDINGS отдельная находка по MR **пока не вводится**.

### 5. Инструкции ролей

Текст [`business-analyst/INSTRUCTIONS.md`](/home/covex/Calculator/business-analyst/INSTRUCTIONS.md) и [`product-manager/INSTRUCTIONS.md`](/home/covex/Calculator/product-manager/INSTRUCTIONS.md) по итогам этого ADR **не меняется**: достаточность линии ADR-019 / ADR-017 подтверждена совместно с правкой верификатора.

## Обоснование

- Разделение «формулировки BA/PM» и «промпт верификатора» нужно явно: иначе эксперимент ошибочно ужесточал бы `REQ-…` деталями реализации.
- Полный положительный проход требований и задач — существенный рубеж этапов 2–3; его фиксируем в ADR и README, а повторяемые уроки для Helper — в FINDINGS.
- Отказ принимать неудачный вывод по MR предотвращает преждевременную порчу промпта MR.

## Рассмотренные альтернативы

| Альтернатива | Почему отклонена |
| --- | --- |
| Добивать четыре REQ деталями реализации до «зелёного» при старом промпте | Ломает границы роли BA; маскирует дефект верификатора |
| Не фиксировать исход, пока не закрыт MR-промпт | Исход Issue уже воспроизводим и значим сам по себе |
| Сразу вводить F-011 по MR | Вывод экспериментатору не принят; нет устойчивой формулировки находки |
| Менять инструкции BA/PM под старый промпт | Противоречит ADR-019 и цели поведенческих требований |

## Последствия

- README этапов 2 и 3 отражает полный положительный проход верификации Issue (требования и задачи) и ссылку на настоящий ADR / F-010.
- Исторические цифры ADR-011 не переписываются; актуальная линия — ADR-019 + ADR-020 + правка промпта Helper.
- Открытый долг: повторный анализ промпта верификации Merge Request.
- Открытый долг из ADR-011: удобство идентификации `REQ-…` ↔ `#N` — без изменений.
