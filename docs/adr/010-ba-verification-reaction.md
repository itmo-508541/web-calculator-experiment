# ADR-010: Реакция Бизнес-аналитика на верификацию Issue

## Статус

Принято; уточняет инструкцию BA и снимает оговорку [ADR-008](008-pm-verification-reaction.md) («BA без изменений»); правки [`business-analyst/INSTRUCTIONS.md`](/home/covex/Calculator/business-analyst/INSTRUCTIONS.md) внесены (2026-07-14).

## Дата

2026-07-14

## Контекст

В [ADR-008](008-pm-verification-reaction.md) петля «верификация → обсуждение → правка только после явного указания» была введена для Продукт-менеджера; инструкцию BA тогда **не** меняли (ожидался семантический поиск по требованиям в GitLab AI Helper).

Экспериментатор решил пройтись по всем требованиям спецификации через верификацию GitLab AI Helper и хочет, чтобы **Бизнес-аналитик** принимал результат верификации так же, как PM: сначала обсуждение, затем обновление текста только после явного указания Пользователя. Ужесточать первую формулировку «под верификатор» по-прежнему не нужно.

## Решение

### 1. Новый шаг 4 в инструкции BA

После переноса требований в систему хранения (после шага 3):

- разбор верификации — **только если** Пользователь **передал** её результат; замечаний может не быть;
- сначала обсуждение (включая ложные срабатывания и спорные пункты), без немедленной правки;
- обновление `requirements-specification.md` / Приложения B / Issue — **только после явного указания** Пользователя;
- цель переформулирования — чтобы Issue могла быть принята верификацией / считалась готовой;
- при нехватке воли заказчика — запрос к Аккаунт-менеджеру, не догадка;
- границы `REQ-…` vs «Решения анализа» сохраняются;
- цикл повторяется по переданным Issue.

### 2. Без ужесточения первого черновика

Как в ADR-008: в инструкцию **не** добавляются правила «сразу писать сверхдетально под верификатор».

### 3. Отдельный файл обсуждения не вводим

Правила — в `INSTRUCTIONS.md`; обсуждение — в диалоге с Пользователем.

## Обоснование

- Единый HITL-паттерн для BA и PM упрощает прогон верификации корпуса требований.
- Явное указание на обновление отделяет согласование от мутации Issue.
- Триггер «если Пользователь передал» учитывает прогон без замечаний и выборочную проверку Issue.

## Рассмотренные альтернативы

| Альтернатива | Почему отклонена |
| --- | --- |
| Ждать семантический поиск в Helper, не трогать BA | Экспериментатор хочет прогон верификации сейчас |
| Ужесточить формулировки REQ заранее под верификатор | Отклонено тем же принципом, что ADR-008 |
| BA сам правит Issue при любом замечании Helper | Без HITL и явного указания Пользователя |

## Последствия

- После переноса `REQ-…` в GitLab BA готов разбирать переданные вердикты и править формулировки по указанию.
- Оговорка ADR-008 п.4 («BA без изменений») для новых работ **не действует**; исполняемая истина — текущий `business-analyst/INSTRUCTIONS.md` и настоящий ADR.
- Известные ограничения Helper (F-001 … F-009) при прогоне могут снова проявиться; их фиксация — по факту в FINDINGS, не блокирует введение шага 4.
