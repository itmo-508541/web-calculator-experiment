# ADR-009: Ведущий разработчик и отказ PM от выбора стека

## Статус

Принято, заменяет в части выбора стека [ADR-006](006-product-manager-instructions.md) (шаги «ресурсы» и «предложение стека»); уточняет [ADR-007](007-pm-task-trace-roles.md) и [ADR-008](008-pm-verification-reaction.md); инструкция [`product-manager/INSTRUCTIONS.md`](/home/covex/Calculator/product-manager/INSTRUCTIONS.md) и [README.md](../../README.md) обновлены (2026-07-12).

Базовый корпоративный каркас зафиксирован в [`tech-stack.md`](../../tech-stack.md) эксперимента (инвентаризация; инструкция сбора — [`docs/tech-stack-inventory-INSTRUCTIONS.md`](../tech-stack-inventory-INSTRUCTIONS.md)). Инструкция этапа Ведущего разработчика — отдельным решением / позже.

## Дата

2026-07-12

## Контекст

Первый прогон этапа 3 показал: Продукт-менеджер (ИИ) выбирал технологический стек после опроса «ресурсов». На практике PM не является носителем этой компетенции; в компании есть репозиторий «пустого приложения» с Docker, CI/CD, hardened-контейнерами и типовым набором модулей.

Согласовано с экспериментатором:

- ввести этап **подготовки окружения** и роль **Ведущий разработчик** (каталог `/home/covex/Calculator/senior-developer`);
- PM **не** выбирает язык, СУБД, фреймворк и т.п.;
- первая работа PM при отсутствии стека — **задача Ведущему** (класс потребностей продукта + ограничения Заказчика + «запустить на мощностях компании»; сервер выбирает Ведущий);
- Ведущий адаптирует базовый каркас, отдаёт PM актуальный `tech-stack.md` (это правда PM о стеке);
- позже возможны повторные задачи Ведущему на **изменение** стека (PM может вернуть свой `tech-stack.md`);
- прогон PM с выбором Slim и связанные Issue **сбрасываются**; этап начинается с чистого листа.

## Решение

### 1. Роль и этап «Ведущий разработчик»

| Аспект | Решение |
| --- | --- |
| Каталог | `/home/covex/Calculator/senior-developer` |
| Акторы | Пользователь = Ведущий разработчик; ИИ-ассистент помогает Ведущему (инструкция этапа — TBD) |
| Вход | Базовый `tech-stack.md` (корпоративный каркас) + номер задачи Issue от PM |
| Работа | Адаптация модулей/пакетов и репозитория под задачу; обновление `tech-stack.md`; список работ по приведению системы в требуемое состояние |
| Выход для PM | Адаптированный `tech-stack.md` |
| Способ переноса каркаса в product-repo | На усмотрение Ведущего (fork, копирование, история коммитов и т.п.) — вне акцента эксперимента |

В README экспериментa этапы перенумерованы: после BA — PM (3), подготовка окружения (4), разработка (5).

### 2. Инструкция Продукт-менеджера (новая логика шагов)

| Было (ADR-006) | Стало |
| --- | --- |
| Шаг 1: опрос серверных ресурсов | Шаг 1: обеспечить `tech-stack.md` через задачу **Ведущему**, если файла нет |
| Шаг 2: PM предлагает и утверждает стек | Удалено; стек не предлагает |
| Шаг 3: волны задач Разработчику | Шаг 2: волны; адресат — Ведущий или Разработчик |
| `tech-stack.md` пишет PM | `tech-stack.md` только **принимает** от Ведущего (через Пользователя) |

Сохраняются: трассировка `REQ-… #<номер>` с ролями «выполнить» / «не нарушить» (ADR-007); реакция на верификацию Issue только после передачи результата и правки только после явного указания (ADR-008) — для задач и Ведущему, и Разработчику.

В `tasks-list.md` добавляется колонка **Адресат**.

### 3. Содержание задачи Ведущему

PM описывает **класс потребностей** и ограничения (в т.ч. домен Заказчика, запуск на мощностях компании), ссылается на `REQ-… #<номер>`, **не** назначает конкретные технологии. Первая задача — на весь каркас, готовый к задачам Разработчика. Повторы — при пробелах или смене стека.

### 4. Базовый `tech-stack.md` в эксперименте

Инвентаризация корпоративного каркаса хранится в проекте эксперимента как [`tech-stack.md`](../../tech-stack.md) (одно имя файла на всех этапах передачи; отдельные `*.baseline.md` не вводятся). Копия/адаптация у Ведущего и снимок у PM — по процессу передачи Пользователем.

## Обоснование

- Выбор стека и подготовка окружения — инженерная ответственность; PM отвечает за приоритеты и формулировку **что** нужно продукту.
- Корпоративный каркас снижает риск «придумать стек с нуля» и дублирования CI/CD/SSL в задачах вайб-кодера.
- Повторные задачи Ведущему отражают реальные пробелы («забыли nginx») без возврата выбора технологий PM.
- Явный гейт «нет `tech-stack.md` → нет волн Разработчику» предотвращает повтор первого прогона.

## Рассмотренные альтернативы

| Альтернатива | Почему отклонена |
| --- | --- |
| PM по-прежнему выбирает стек после опроса ресурсов | Не соответствует компетенции роли; отвергнуто после первого прогона |
| Ведущий без директории и без ИИ | Отвергнуто: нужна рабочая директория и помощь ИИ при адаптации каркаса |
| `tech-stack.baseline.md` отдельно от `tech-stack.md` | Отвергнуто: одно имя файла везде |
| Окружение готовит только Разработчик (вайб-кодер) в этапе разработки | Слишком поздно и смешивает каркас с продуктовой логикой |

## Последствия

- Первый прогон PM (Slim, волны Compose и т.д.) не продолжается; экспериментатор удаляет Issue и артефакты PM и начинает заново по новой инструкции.
- Инструкцию `senior-developer/INSTRUCTIONS.md` ещё нужно подготовить отдельным шагом.
- ADR-006 остаётся историческим описанием прежней модели; исполняемая истина — текущий `product-manager/INSTRUCTIONS.md` и настоящий ADR.
