# ADR-011: Смена линии BA — превентивное качество формулировок после прогона верификации

## Статус

Принято как **смена линии** для следующей правки инструкции Бизнес-аналитика; частично меняет установку [ADR-008](008-pm-verification-reaction.md) / [ADR-010](010-ba-verification-reaction.md) («не ужесточать первый черновик под верификатор») **в части BA**. Текущий [`business-analyst/INSTRUCTIONS.md`](/home/covex/Calculator/business-analyst/INSTRUCTIONS.md) **пока не меняется** — шаг 4 (реактивная правка) остаётся в силе до отдельной итерации.

FINDINGS не дополняется: итоги прогона и процессная линия эксперимента фиксируются здесь, не в документе для GitLab AI Helper.

## Дата

2026-07-15

## Контекст

После [ADR-010](010-ba-verification-reaction.md) экспериментатор прогнал через верификацию GitLab AI Helper **все** требования спецификации, перенесённые в GitLab Issue.

### Итоги прогона

| Показатель | Значение |
| --- | --- |
| Всего требований в прогоне | 29 |
| Прошли верификацию (итоговый вердикт «готово») | 17 |
| Не прошли / не доведены до «готово» | 12 |
| Прошли **с первого** прогона | 0 |

Цикл «Issue → верификация → обсуждение с BA → правка по явному указанию → повтор» оказался **рабочим**, но **дорогим**: ни одно требование не закрылось без хотя бы одной итерации доработки формулировки.

В ADR-008/010 сознательно **не** закладывали превентивную «сверхдетализацию под верификатор», чтобы не раздувать первый черновик. Прогон показал: при текущей инструкции почти весь корпус требует реактивной переформулировки.

### Дополнительное наблюдение (отложено)

Неудобно, что одно и то же требование имеет разные идентификаторы: во внутренних артефактах BA — `REQ-005`, в системе хранения — `#10` (и аналогично для остальных). Единая схема именования / отображения **пока не решается**; к ней вернуться позже.

## Решение

### 1. Смена линии для следующей правки инструкции BA

При **следующем** изменении [`business-analyst/INSTRUCTIONS.md`](/home/covex/Calculator/business-analyst/INSTRUCTIONS.md) заложить, чтобы Бизнес-аналитик **изначально** формировал требования, обладающие свойствами качества (однозначность, полнота описания отдельной Issue, непротиворечивость, проверяемость и связанные с ними критерии продукта верификации), а не рассчитывал только на шаг 4 после прогона Helper.

Допустимый (но не единственный) способ — привлечение **суб-агентов** или иного внутреннего прохода проверки формулировок **до** передачи Пользователю на перенос в GitLab. Конкретный дизайн инструкции и инструментов — в той итерации; настоящий ADR фиксирует **намерение и причину**, не текст инструкции.

### 2. Реактивный шаг 4 не отменяется сразу

Пока инструкция не переписана:

- шаг 4 ADR-010 остаётся исполняемой нормой;
- прогон корпуса для текущей спецификации считается **завершённым** с результатом 17/29;
- дальнейшие правки отдельных Issue — по необходимости экспериментатора, не как продолжение обязательного полного прогона.

### 3. Идентификаторы `REQ-…` vs `#N`

Зафиксировать проблему как **открытый вопрос** эксперимента. Решение (единый ID, отображение в заголовке Issue, поддержка в Helper и т.п.) — отдельно, не в этом ADR и не в FINDINGS без связи с конкретным разрывом продукта Helper.

## Обоснование

- Нулевое число «с первого раза» при полном прогоне — сигнал, что реактивная линия недостаточна для приемлемой стоимости этапа BA.
- Смена линии относится к **инструкциям эксперимента** (как писать REQ), а не к дефекту GitLab AI Helper; потому ADR, не FINDINGS.
- Отложенное изменение файла инструкции избегает смешивания «зафиксировали итог прогона» и «уже внедрили новый процесс» без отдельной итерации текста.

## Рассмотренные альтернативы

| Альтернатива | Почему отклонена |
| --- | --- |
| Оставить только реактивный шаг 4 | Прогон показал неприемлемую долю итераций на каждое REQ |
| Сразу переписать INSTRUCTIONS.md в этом же шаге | Экспериментатор зафиксировал линию «на следующий раз», не сиюминутную правку |
| Записать итоги только в FINDINGS | FINDINGS — для развития GitLab AI Helper; здесь — процесс эксперимента и инструкция BA |
| Сейчас же менять схему `REQ-…` / `#N` | Явно отложено («подумать позже») |

## Последствия

- Следующая правка инструкции BA должна реализовать превентивное обеспечение свойств формулировок (в т.ч. с возможным использованием суб-агентов) и согласоваться с этим ADR; при конфликте с формулировками ADR-008/010 про «не ужесточать первый черновик» для **BA** приоритет у настоящего ADR.
- Для **PM** установка ADR-008 (не ужесточать первый черновик задач) этим решением **не** отменяется.
- README этапа 2 отражает завершение прогона верификации и ссылку на ADR-011.
- Открытый долг: удобство идентификации `REQ-…` ↔ `#N`.
