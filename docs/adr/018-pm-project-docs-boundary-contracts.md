# ADR-018: PM — проектная документация и контракты границ

## Статус

Принято; уточняет [ADR-009](009-senior-developer-and-pm-stack.md), [ADR-014](014-lead-dev-module-recommendations.md) и [ADR-017](017-pm-proactive-task-quality.md); правки [`product-manager/INSTRUCTIONS.md`](/home/covex/Calculator/product-manager/INSTRUCTIONS.md) и [README.md](../../README.md) внесены (2026-07-17).

## Дата

2026-07-17

## Контекст

После появления стека прикладные задачи Разработчику формулировались без зафиксированного соглашения на стыке частей системы (например контракта API). Верификация справедливо требовала ясности; пришлось вне плана ставить задачу Ведущему на фиксацию такого соглашения и только затем опираться на него в задачах реализации.

Обобщение: окружение/`tech-stack.md` и подобные соглашения — один класс **контрактов границ**. Принцип «сначала контракт — потом работа в рамках контракта» нельзя откладывать: пропуск основания ломает зависимые задачи (как ранее с неверифицированными требованиями).

Нужен единый каталог снимков артефактов Ведущего у PM и индекс для ИИ-ассистента.

## Решение

### 1. Проектная документация `docs/`

- Каталог `docs/` у PM — **проектная документация**: копии документов, которые Пользователь получил от Ведущего.
- PM ведёт `docs/README.md` — индекс доступных документов в удобном для себя виде (жёсткий шаблон не требуется).
- `docs/tech-stack.md` — обязательный минимум для перехода к волнам Разработчику (шаг 1). Путь — в `docs/`, не в корне рабочей директории PM.

### 2. Контракты границ

**Контракт границ** — зафиксированное соглашение на стыке частей системы (или системы и внешнего мира), от которого зависят несколько последующих задач.

- Стек/окружение — контракт границ того же класса для старта разработки.
- Иные контракты границ — отдельные задачи по мере надобности волны.
- **Жёстко:** сначала контракт, потом задачи реализации в его рамках; не смешивать «создать контракт» и «реализовать фичу»; не смешивать окружение и иной контракт в одной Issue.
- Адресат задач на стек, окружение и контракты границ — **всегда Ведущий разработчик**.
- Результат: изменения в репозитории продукта; дополнительно документ передаётся PM в `docs/`.

### 3. Гейт для задач Разработчику

Не ставить задачу Разработчику, если для однозначности и проверяемости ещё нет нужного контракта границ в `docs/`. Обнаруженный при верификации/формулировании пробел → задача Ведущему, не додумывание контракта в тексте Issue Разработчика.

Инструкция **не** ссылается на конкретные имена файлов контрактов продукта (кроме `tech-stack.md` на шаге 1) и не привязана к конкретному продукту эксперимента.

## Обоснование

- Фундамент (каркас, контракты границ) — зона компетенции Ведущего; PM не выбирает и не изобретает границы.
- Дробление снижает неоднозначность и шум верификации.
- `docs/` + индекс дают PM устойчивый вход для постановки без опоры на «память чата».

## Рассмотренные альтернативы

| Альтернатива | Почему отклонена |
| --- | --- |
| Контракт внутри задачи Разработчику | Ломает принцип; неоднозначность и верификация |
| Адресат контракта — иногда Разработчик | Отклонено: всегда Ведущий |
| Жёсткий шаблон `docs/README.md` | Достаточно удобства для ИИ-ассистента PM |
| Упоминать конкретный файл контракта API в инструкции | Инструкция должна быть проект-агностичной |

## Последствия
