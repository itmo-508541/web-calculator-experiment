# ADR-003: Принятие этапа 1 и этап 2 (Бизнес-аналитик)

## Статус

Принято (объединяет ADR-003…007 от 2026-07-08 в один документ)

## Дата

2026-07-08

## Контекст

Первый прогон этапа «Сбор информации от заказчика» завершён по [ADR-002](002-roles-and-account-manager.md). Экспериментатор выступал в роли Заказчика, ИИ-ассистент — в роли Аккаунт-менеджера. Итоговый артефакт — [`customer-elicitation.md`](/home/covex/Calculator/account-manager/customer-elicitation.md).

Экспериментатор **принял результат** этапа 1 («в целом результат меня устроил»). Инструкция этапа 2 в README была помечена TBD до принятия работы Аккаунт-менеджера.

### Наблюдения по артефакту этапа 1

- Трёхшаговая схема (анализ → план → интервью) с HITL отработала: все четыре раздела `customer-elicitation.md`, протокол подтверждён Заказчиком.
- Раздел «Собранная информация» пригоден для BA/PM без превращения в спецификацию.
- Закрыты основные пробелы исходного PDF; зафиксированы приоритеты, границы v1, эталон.
- Остаются **открытые вопросы** (раздел 10 артефакта) — часть делегирована BA, часть потребует петли уточнения через Аккаунт-менеджера.

### Что не меняем в инструкции Аккаунт-менеджера

Принятие этапа не требует правок [`account-manager/INSTRUCTIONS.md`](/home/covex/Calculator/account-manager/INSTRUCTIONS.md). Итерация успешна; доработки — по явному запросу экспериментатора или после первого реального запроса от BA (см. ниже).

## Решение

### 1. Этап 1 — принят

Этап «Сбор информации от заказчика» **успешно завершён** (2026-07-08). Вход для этапа 2 — `customer-elicitation.md`, который **передаёт Пользователь** Бизнес-аналитику при запуске.

### 2. Инструкция Бизнес-аналитика

Файл [`business-analyst/INSTRUCTIONS.md`](/home/covex/Calculator/business-analyst/INSTRUCTIONS.md).

| Аспект | Решение |
| --- | --- |
| Роль | **Анализ требований** — сходящий процесс: уточнение, устранение противоречий, приоритизация; результат — спецификация ([глава 05](../requirements-engineering/05-analiz-trebovaniy.md)) |
| Изоляция | ИИ-ассистент **не знает** об эксперименте; в инструкции **нет** предметной области, ссылок на `experimentator`, `REQUIREMENT.md`, конспект лекции |
| Собеседник | **Пользователь** — оценивает HITL, переносит требования в систему хранения, передаёт запросы AM |
| Вход | **Только** файл, который передаёт Пользователь (результат этапа сбора информации). **Не** исходный PDF/иллюстрации Заказчика |
| Итог | `requirements-specification.md` |
| Структура спецификации | Введение, общее описание, специфические требования (`REQ-…`), критерии приёмки, приложение A (сценарии проверки), приложение B (тексты для системы хранения) |
| IEEE 830 | Все восемь свойств — **критерии формулировки** каждого требования |
| Приоритеты | Класс: необходимое / условное / необязательное |
| HITL | Три шага: анализ и план → черновик → финализация и handoff |
| Границы | BA не создаёт Issue, не формирует задачи разработки, не общается с Заказчиком напрямую |
| TBD в требованиях | **Запрещены** в принятых REQ; нерешённое — через запросы к AM |

### 3. Запросы Бизнес-аналитика к Аккаунт-менеджеру

Если BA нужна **любая** информация, которой нет во входном материале (уточнение у Заказчика, детали оформления, разрешение противоречия и т.п.):

- **Один запрос — один файл:** `account-manager-request-001.md`, `account-manager-request-002.md`, …
- Нумерация в **имени файла** (три цифры); сквозная в каталоге `business-analyst/`. Перед новым запросом BA смотрит существующие `account-manager-request-*.md`.
- Содержание файла: запрос, контекст, зачем нужно, влияние на `REQ-…`.
- Цепочка: **BA → Пользователь → Аккаунт-менеджер** (не BA → Заказчик, не BA → PDF).
- Если запросов нет — файлы не создаются.

### 4. README experimentator

- Этап 1: статус «завершён», ссылка на `customer-elicitation.md`.
- Этап 2: ссылка на инструкцию BA; вход и результаты; петля уточнений BA → User → AM.

## Обоснование

**Инструкции по мере принятия** (ADR-002): этап 1 принят — готовим этап 2; инструкция AM не трогаем без необходимости.

**Изоляция инструкции BA.** Эксперимент проверяет No/Low-coding процесс; утечка контекста эксперимента или предметной области в инструкцию искажает прогон.

**Разделение артефактов.** AM выдаёт материал о пожеланиях; BA получает его от Пользователя. Предметная область — только во входном файле.

**PDF не передаётся BA.** Первичные артефакты — зона AM; BA работает с текстовой выжимкой. Нехватка данных — запрос AM, не обход через Пользователя.

**Отдельный файл на запрос.** Упрощает передачу Пользователем и отслеживание итераций; один запрос — один артефакт.

**Отложенная инструкция AM.** Как AM обрабатывает `account-manager-request-NNN.md`, как обновляется `customer-elicitation.md`, как устроен повторный прогон BA — **решаем после первого реального запроса**, не проектируем заранее.

## Рассмотренные альтернативы

| Альтернатива | Почему не выбрана |
| --- | --- |
| BA возвращается к Заказчику по каждой детали | Заказчик делегировал детализацию BA; граница ролей |
| BA создаёт Issue через MCP | Противоречит роли Пользователя на этапе 2 |
| Ссылки на experimentator в инструкции BA | Искажает эксперимент |
| Раздел «Требует согласования» внутри спецификации | Усложняет передачу между ролями |
| Единый файл `account-manager-requests.md` с ID `AMR-…` | Заменено на один файл на запрос |
| Передача PDF BA напрямую | Нарушает разделение ролей AM / BA |
