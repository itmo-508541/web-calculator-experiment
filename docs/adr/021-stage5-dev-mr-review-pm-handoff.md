# ADR-021: Этап 5 — разработка, Code Review, сдача Продукт-менеджеру

## Статус

Принято; дополняет [ADR-009](009-senior-developer-and-pm-stack.md) (этап разработки) и уточняет шаг **2.D** инструкции Продукт-менеджера ([ADR-006](006-product-manager-instructions.md) и последующие). Вводит инструкцию ИИ-ассистента этапа 5. Обновлены [README.md](../../README.md), [`docs/developer/INSTRUCTIONS.md`](../developer/INSTRUCTIONS.md), [`product-manager/INSTRUCTIONS.md`](/home/covex/Calculator/product-manager/INSTRUCTIONS.md) (2026-07-18).

## Дата

2026-07-18

## Контекст

Этап 5 в README оставался TBD. В эксперименте впервые появился настоящий Merge Request с кодом по задаче; система верификации MR (GitLab AI Helper) дала положительный отзыв. Дальнейший процесс не был определён: задача никак не сдавалась Продукт-менеджеру, петля Code Review не была разыграна.

В компании (эталон процесса):

1. Разработчик создаёт Merge Request.
2. Ведущий разработчик проводит Code Review (замечания в MR / к коду).
3. Разработчик устраняет замечания и снова предлагает изменения; цикл до отсутствия недочётов.
4. Ведущий принимает MR.
5. Разработчик сообщает Продукт-менеджеру, что задача готова.
6. PM организует QA (детали вне фокуса экспериментатора).

Цели GitLab AI Helper (ускорить ревью MR, снизить итерации постановки, повысить прозрачность) согласуются с тем, чтобы **отработать** ревью и сдачу на реальном MR, не подменяя Code Review и продуктовую приёмку одним вердиктом бота.

Согласовано с экспериментатором: смоделировать корпоративный процесс в ролях эксперимента; полную организацию QA не воспроизводить — только видимые недочёты → новая задача в текущей волне.

## Решение

### 1. Этап 5 в README и инструкция

| Аспект | Решение |
| --- | --- |
| Каталог кода | `/home/covex/Calculator/developer` (репозиторий продукта) |
| Инструкция ИИ | [`docs/developer/INSTRUCTIONS.md`](../developer/INSTRUCTIONS.md) в репозитории эксперимента (не коммитить процессный файл в продукт) |
| Акторы в чате разработки | Пользователь = **Вайб-кодер** (Разработчик); ИИ = помощник по реализации задачи |
| Code Review | Пользователь **вне** чата разработки выступает как **Ведущий разработчик**: смотрит MR, готовит текст ситуации / замечаний; затем в чате разработки передаёт этот текст ИИ |
| Принятие MR | Делает Пользователь как Ведущий (merge / accept в GitLab); ИИ этапа 5 merge не выполняет сам |
| Сдача PM | После принятия MR Пользователь **сообщает** Продукт-менеджеру, что задача готова (пакет: номер Issue, ссылка на MR, кратко что сделано / как проверить) |

Отдельная инструкция ИИ для Code Review на этапе Ведущего **не** вводится: ревью готовит Пользователь текстом. Инструкция этапа 4 (`senior-developer`) без изменений.

### 2. Шаги для ИИ-ассистента Разработчика (этап 5)

1. **Реализация** — по номеру Issue (MCP) и репозиторию продукта выполнить задачу; критерии и границы — из текста Issue и `docs/` продукта / переданного Пользователем.
2. **Merge Request** — подготовить изменения и MR (или помочь Пользователю это сделать); остановиться на HITL: «MR готов к Code Review».
3. **Реакция на Code Review** — разбирать замечания **только если** Пользователь **передал** текст от Ведущего (или эквивалент: выжимка комментариев MR). Внести правки в код / MR; снова HITL. Цикл до сообщения Пользователя, что Ведущий **принял** MR.
4. **Handoff** — после принятия MR кратко собрать пакет для передачи Пользователем Продукт-менеджеру; **не** обращаться к PM напрямую; **не** начинать следующую задачу без указания Пользователя.

Верификация MR ботом Helper — **входной сигнал**, не замена Code Review и не замена приёмки PM. Разбор комментариев Helper — только если Пользователь их передал (по аналогии с ADR-008).

### 3. Уточнение шага 2.D у Продукт-менеджера

После готовности волны к выполнению PM ждёт не «абстрактный отчёт», а сообщение Пользователя, что **задача(и)** готовы **после принятия MR Ведущим**.

- Вместе с Пользователем сверить результат с **критериями готовности** задачи и связанными требованиями (роли «выполнить» / «не нарушить»).
- Автоверификация MR **не** считается приёмкой задачи.
- Если Пользователь сообщает о **видимых недочётах** (без организации полного QA) — **не** откатывать уже принятый MR в инструкции PM; сформулировать **новую задачу на исправление** в рамках **текущей** волны (HITL → перенос в Issue как обычно), затем снова ожидание выполнения.
- Если недочётов нет / Пользователь подтвердил приёмку — зафиксировать в `task-plan.md` и переходить к 2.E.

Единица сдачи: по умолчанию **отдельная Issue** (атомарные задачи волны); волна закрыта в плане, когда приняты нужные задачи волны (и исправления, если были).

### 4. Роли Пользователя на этапе 5 (логирование)

Один человек последовательно:

| Момент | Роль Пользователя |
| --- | --- |
| Реализация и правки по ревью в чате с ИИ | Вайб-кодер |
| Осмотр MR, текст замечаний, принятие MR | Ведущий разработчик |
| Передача «задача готова» / видимых недочётов в чат PM | Пользователь (курьер к PM) |

Действия логируются для анализа эксперимента (как и на этапах 2–4).

### 5. Связь с GitLab AI Helper

Эксперимент на этом этапе отрабатывает цепочку, ради которой Helper ускоряет ревью и прозрачность MR. Правки промпта верификации MR по-прежнему **отложены** ([ADR-020](020-full-issue-verification-pass.md)); находки по факту первого живого ревью — в FINDINGS отдельной итерацией, если проявятся.

## Обоснование

- Корпоративный процесс уже разделяет Code Review (Ведущий) и продуктовую готовность (PM); эксперимент должен воспроизвести это, а не схлопнуть в «зелёный бот → закрыто».
- HITL и курьерство Пользователя согласуются с остальными этапами: ИИ ролей не ходят друг к другу напрямую; MCP не отдаёт MR.
- Исправления видимых недочётов **новой задачей** в волне сохраняют итеративность PM и не смешивают приёмку с повторным Code Review того же MR без явной нужды.
- Инструкция этапа 5 вне git продукта исключает засорение репозитория калькулятора процессными файлами эксперимента.

## Рассмотренные альтернативы

| Альтернатива | Почему отклонена |
| --- | --- |
| Зелёный вердикт Helper = приёмка / merge | Подменяет Code Review и PM; промпт MR ещё не принят (ADR-020) |
| ИИ Ведущего сам делает Code Review по diff | Экспериментатор хочет сначала увидеть живой ручной ревью и передать текст |
| Полный процесс QA у PM | Вне компетенции/фокуса экспериментатора; достаточно видимых недочётов → задача |
| Класть `INSTRUCTIONS.md` в корень продукта | Риск попадания в историю продукта; источник истины — репозиторий эксперимента |
| PM правит уже принятую задачу вместо новой Issue на фикс | Ломает след «что было сдано»; новая задача в волне проще для трассировки |
