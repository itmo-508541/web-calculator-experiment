# ADR-016: PM — гранулярность волны и ссылки между задачами

## Статус

Принято; уточняет [ADR-006](006-product-manager-instructions.md), [ADR-007](007-pm-task-trace-roles.md) и [ADR-013](013-pm-storage-refs.md); правки [`product-manager/INSTRUCTIONS.md`](/home/covex/Calculator/product-manager/INSTRUCTIONS.md) внесены (2026-07-15).

## Дата

2026-07-15

## Контекст

После появления `tech-stack.md` Продукт-менеджер сформулировал первую волну задач для Разработчика как **одну** крупную Issue, которую экспериментатор затем попросил разбить на более мелкие. Отдельно: ссылки между задачами в тексте Issue («связана с», «зависит от», «блокирует») дают шум системе верификации; порядок и зависимости Пользователь передаёт Разработчику сам.

## Решение

### 1. Гранулярность волны (мягкое правило)

При формулировании ближайшей волны PM **предпочитает** несколько относительно атомарных Issue, а не одну задачу «на всю волну». Объединение смысловых частей волны в одну Issue — **только** по явной просьбе Пользователя.

Порядок и блокеры между работами по-прежнему учитываются в `task-plan.md` и при выборе ближайшей волны.

### 2. Ссылки на другие задачи в тексте Issue

Ссылка на другую задачу в черновике / теле Issue **допускается только**, если текущая задача **дополняет** ту, на которую ссылается.

**Не** добавлять в текст задачи формулировки и ссылки вида «связана с», «зависит от», «блокирует» / «разблокирует» и аналогичный граф зависимостей через `#…`.

Такую информацию Пользователь передаёт Разработчику вне текста Issue. Зависимости для планирования — в `task-plan.md`, не в описании задач.

Трассировка к **требованиям** (`#…` с ролями **выполнить** / **не нарушить**) без изменений.

## Обоснование

- Одна Issue на всю волну усложняет выполнение и приёмку; дробление по умолчанию ближе к фактической практике эксперимента.
- Межзадачные ссылки «зависимости» пересекаются с проверками верификатора и дублируют то, что Пользователь и так сообщает Разработчику при выдаче работ.

## Рассмотренные альтернативы

| Альтернатива | Почему отклонена |
| --- | --- |
| Жёсткий запрет любой одной «крупной» Issue | Экспериментатор выбрал мягкое правило: объединять можно по просьбе Пользователя |
| Оставить блок «Зависимости» в тексте задачи | Шум для верификации; порядок передаёт Пользователь |

## Последствия

- Новые волны задач формулируются с предпочтением дробления.
- В уже созданных Issue при следующей правке по указанию Пользователя убирать лишние межзадачные ссылки, оставляя только «дополняет» при необходимости.
