## Контекст

После ADR-021 инструкция этапа 5 жила одним файлом в репозитории эксперимента; Разработчик должен был открывать её вручную. На практике в окружении продукта уже использовалась Cursor-команда `/task` (загрузка Issue, уточнения, реализация, QA через skills/Docker). Отдельно потребовалось фиксировать решения реализации в ADR продукта — чтобы облегчить Code Review Ведущему — **после** кода и тестов, отдельным шагом.

Согласовано с экспериментатором:

- две команды: `/task <номер>` и `/adr`;
- типичный порядок: сначала задача, затем ADR; в итоге — изменения кода и файл ADR, затем MR / ревью / сдача PM как в ADR-021;
- номер ADR продукта — **сквозной хронологический**, не номер Issue (иначе ломается порядок решений);
- статус **Принято** в ADR продукта — про зафиксированное решение, не про accept MR (Ведущий не обязан менять статус отдельным коммитом).

## Решение

### 1. Две Cursor-команды в окружении Разработчика

| Команда | Назначение |
| --- | --- |
| `/task <номер>` | Роль ИИ Разработчика; загрузка Issue (MCP); уточнения и HITL до явного подтверждения; реализация; обязательные тесты; QA-прогон (`make cs` / `phpstan` / `rector` / `test` / `behat`). **Не** создаёт ADR. |
| `/adr` | Один ADR по уже выполненной задаче текущего чата; для Code Review. **Не** реализует новую функциональность вместо `/task`. |

Файлы команд: `/home/covex/Calculator/developer/.cursor/commands/task.md` и `adr.md`.

### 2. Источник текстов в эксперименте

| Файл | Роль |
| --- | --- |
| [`docs/developer/task.md`](../developer/task.md) | Полный текст команды `/task` (инструменты + роли + шаги; MR / Code Review / сдача PM — раздел «Далее», по указанию Пользователя) |
| [`docs/developer/adr.md`](../developer/adr.md) | Полный текст команды `/adr` (именование, статусы, структура, правила) |
| [`docs/developer/INSTRUCTIONS.md`](../developer/INSTRUCTIONS.md) | Только **указатель** на две команды и порядок запуска |

При правке инструкции обновлять **и** файлы в эксперименте, **и** копии в `.cursor/commands/` продукта.

### 3. Порядок работы

1. `/task N` — до зелёного QA (в разумных пределах задачи) и HITL «запустите `/adr`».
2. `/adr` — файл в `docs/adr/` продукта.
3. Далее по ADR-021 / разделу «Далее» в `task.md`: MR → Code Review (текст от Ведущего) → accept MR → пакет сдачи PM.

### 4. ADR продукта (внутри `/adr`)

- Каталог: `docs/adr/` репозитория продукта.
- Имя: `{NNN}-краткое-название.md` — `{NNN}` следующий свободный хронологический номер (`001`, `002`…); номер Issue — в разделе «Контекст».
- Один ADR на задачу (несколько решений — разделы одного файла).
- Писать **после** реализации и прохождения тестов/QA (`/task`); к Code Review статус обычно **Принято**.
- **Черновик** — допустимый, не обязательный статус.
- Замена решения другой задачей — новый ADR со следующим `{NNN}` («заменяет» / «дополняет» предыдущий); старый файл не удалять.
- Текст команды `/adr` **не** описывает эксперимент: ИИ Разработчика о нём не знает.

Процессный ADR эксперимента (этот каталог `docs/adr/`) и ADR продукта — разные контуры и нумерации.

### 5. Что из ADR-021 остаётся без изменений

Роли Пользователя (Вайб-кодер / Ведущий / курьер к PM); Code Review только по переданному тексту; ИИ не мержит MR; сдача и 2.D PM; Helper MR — сигнал, не приёмка.

## Обоснование

- Slash-команды — естественный HITL-вход в Cursor: Разработчик явно запускает этапы, а не держит длинную моноинструкцию в чате.
- Разделение `/task` и `/adr` отделяет «сделать» от «обосновать для ревью» и совпадает с моментом, когда решения уже воплощены в коде.
- Хронологическая нумерация ADR продукта сохраняет историю решений; Issue остаётся трассировкой, не ключом файла.
- Статус **Принято** без правки после merge убирает лишний коммит Ведущего и не смешивает решение с accept MR.
- Копии в `.cursor/commands/` нужны для исполнения; зеркало в эксперименте — для наблюдения и правок процесса.

## Рассмотренные альтернативы

| Альтернатива | Почему отклонена |
| --- | --- |
| Один `INSTRUCTIONS.md` как единственный вход (ADR-021) | На практике уже была команда `/task`; монофайл неудобен для запуска |
| Номер ADR = номер Issue | Ломает хронологический порядок файлов в `docs/adr/` |
| ADR пишет `/task` сам в конце QA | Смешивает реализацию и обоснование; экспериментатор вынес отдельную команду |
| Статус «Принято» только после merge Ведущим | Требует отдельного коммита Ведущего; путает решение и accept MR |
| Держать тексты команд только в продукте | Эксперимент теряет источник истины для инструкций этапа |

## Последствия
