# ADR-012: Идентификаторы в системе хранения — заголовок `REQ-…` и ссылки `#N`

## Статус

Принято; правки [`business-analyst/INSTRUCTIONS.md`](/home/covex/Calculator/business-analyst/INSTRUCTIONS.md) внесены (2026-07-15). Связано с открытым вопросом [ADR-011](011-ba-proactive-quality-after-verification-run.md) про `REQ-…` ↔ `#N`; фиксирует принятую на прогоне практику оформления Issue.

## Дата

2026-07-15

## Контекст

После переноса требований в GitLab у каждого `REQ-…` появляется номер Issue (`#M`), фиксируемый в `requirements-list.md`. В диалоге с BA:

1. **Заголовок Issue** включает внутренний номер требования — пример загруженной Issue **#26**: заголовок `REQ-025 Вставка из буфера`.
2. **Тексты** Приложения B / описания Issue уже приводились к виду, где отсылки на другие требования — `#M` (например в теле #26: `#8`, `#7`, `#9`), а не `REQ-007`.

Это нужно для поиска по стабильному `REQ-…` в списке Issue и для нативной связности GitLab по `#N` в теле.

Инструкция BA до фиксации ADR не обязывала ни формат заголовка, ни замену отсылок после заполнения таблицы соответствий.

## Решение

### 1. Заголовок требования в системе хранения

Формат **title** Issue при переносе из Приложения B:

```text
REQ-NNN <Краткий заголовок>
```

- `REQ-NNN` — стабильный идентификатор из спецификации (три цифры с ведущими нулями, как `REQ-025`);
- один пробел;
- краткий заголовок **без** повторного префикса `REQ-…` и **без** обязательного тире/`—` между id и текстом (пример факта: `REQ-025 Вставка из буфера`, Issue `#26`).

Тот же заголовок — в колонке «Заголовок» таблицы `requirements-list.md` **вместе с** пониманием, что в GitLab title = `REQ-NNN` + пробел + эта краткая фраза. (В таблице по-прежнему отдельные колонки «Требование» = `REQ-025` и «Заголовок» = краткая фраза; при создании Issue BA/Пользователь собирают title по правилу выше.)

### 2. Перекрёстные ссылки в теле

1. **До переноса** в Приложении B допустимы перекрёстные отсылки `REQ-…`.
2. **После** заполнения `#<номер>` в `requirements-list.md` BA **обязан** заменить в текстах Приложения B отсылки `REQ-…` на `#<номер>` по этой таблице (источник соответствия — `requirements-list.md`).
3. Стабильный id в **заголовке** пункта Приложения B / Issue (`REQ-025 …`) сохраняется; меняются отсылки **внутри** формулировки и проверки.
4. При обновлении Issue (в т.ч. на шаге 4) в теле Issue не возвращать отсылки `REQ-…` на другие требования.

## Обоснование

- Заголовок с `REQ-…` закрывает часть боли «в файле REQ-005, в GitLab #10» на уровне списка Issue, не требуя равенства номеров.
- В теле GitLab нативная навигация и верификация опираются на `#N`.
- Таблица соответствий уже есть у BA — дублировать маппинг не нужно.
- Разделение «до / после номеров» для ссылок в теле не блокирует черновик до переноса.

## Рассмотренные альтернативы

| Альтернатива | Почему отклонена |
| --- | --- |
| Оставлять в Issue только `REQ-…` в теле | Неудобно в системе хранения; расходится с уже принятой практикой прогона |
| Заголовок без `REQ-…` (только «Вставка из буфера») | Теряется стабильный id в UI GitLab |
| Единый id везде (`REQ` = номер GitLab) | Открытый вопрос ADR-011; сейчас не решается |
| Формат с тире: `REQ-025 — Вставка…` | Факт прогона и #26 — пробел без тире |

## Последствия

- После сообщения номеров Пользователем шаг 3 включает обязательную нормализацию ссылок в Приложении B и HITL на этот результат.
- Приложение B и перенос в GitLab используют единый формат title `REQ-NNN <краткий заголовок>`.
- Продукт-менеджер по-прежнему видит `REQ-… #<номер>` в своей трассировке задач.
