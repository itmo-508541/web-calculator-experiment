# /adr — ADR по выполненной задаче

Команда запускается **после** `/task`, когда код реализован и QA-прогон (или согласованный объём тестов) уже пройден. Аргументов нет: опирайся на задачу и изменения **текущего чата** (номер Issue, принятые решения, затронутые файлы). Если контекста задачи нет — спроси номер Issue и при необходимости загрузи её через MCP `load-requirement`.

Создай (или доведи) **один** файл ADR в репозитории. Код и описание MR — про «что сделано». ADR — про **обоснование**: почему сделано именно так. ADR нужен Ведущему для **Code Review**.

В рамках `/adr` **не** реализуй новую функциональность и **не** подменяй `/task`. Правки кода — только если без них нельзя честно описать уже принятое решение (лучше остановись и спроси Пользователя).

## Где и как назвать файл

- Каталог: `docs/adr/`.
- **Один файл на одну задачу** (на один номер Issue), если Пользователь не попросил иначе.
- Имя: `{NNN}-краткое-название.md`, где `{NNN}` — **сквозной хронологический** номер в каталоге (`001`, `002`, `003`…), **не** номер Issue.
  - Посмотри существующие `{NNN}-*.md` и возьми **следующий** свободный номер.
  - Пример: есть `001-…`, `002-…` → новый `003-session-cookie-priority.md` (даже если задача `#32`).
- Краткое название — латиницей, через дефис, по сути решения.
- Номер задачи (`#32` и т.п.) — **внутри** ADR (раздел «Контекст»), не в номере файла.

Номер ADR и номер Issue — разные идентификаторы: порядок решений во времени vs трассировка к задаче.

## Статусы

Статус — про жизненный цикл **решения**, не про merge и не про приёмку Продукт-менеджером.

| Статус | Когда |
| --- | --- |
| **Черновик** | Допустимый промежуточный статус; в типичном потоке может не понадобиться. |
| **Принято** | Основной статус при подготовке ADR к Code Review: решение зафиксировано и отражено в коде. |
| **Принято, заменяет ADR-{NNN}** / **дополняет ADR-{NNN}** | В **новом** ADR (новый хронологический номер), если другая задача меняет или дополняет прежнее решение. |

**Принято** ≠ «MR принят Ведущим» и ≠ «задача закрыта». Ведущий **не** обязан менять статус после merge.

Если позже другая задача меняет решение: новый Issue → новый файл со следующим `{NNN}` и статусом «Принято, заменяет ADR-{старый-NNN}» (или «дополняет»); старый файл не удалять. В контексте укажи номера Issue.

## Зачем (критерий полезности)

Ведущий по ADR должен быстро ответить:

- какую задачу закрывали и в каких границах;
- какие решения приняты в коде и почему;
- что сознательно не делали;
- какие варианты отвергли и чем это грозит дальше.

Не пиши пересказ diff. Если раздел не помогает ревью — сократи.

## Структура файла

Заголовок: `# ADR-{NNN}: краткий заголовок` (тот же `{NNN}`, что в имени файла).

### Статус

Обычно сразу **Принято**. При необходимости — **Черновик**, затем **Принято** до передачи на Code Review.

При замене/дополнении — ссылка на **номер ADR** (`ADR-001`), не на номер Issue.

Дата и ссылка на MR — по желанию, не вместо статуса.

### Дата

YYYY-MM-DD (запись / последнее существенное обновление).

### Контекст

- Issue `#…`, связь с критериями готовности.
- Ограничения: стек, контракты границ, «входит / не входит».
- Развилка или неоднозначность, из‑за которой понадобилось решение.

### Решение

Подход, соглашения, границы. По пунктам или короткой таблицей. Без портянки кода — отсылки к файлам допустимы.

### Обоснование

Почему вариант уместен для этой задачи и ограничений.

### Рассмотренные альтернативы

Альтернатива → почему отклонена (или отложена). Если развилки не было — явно напиши, что решение следует из Issue/контракта.

### Последствия

Поведение/код; что проверить ревьюеру; долги и компромиссы; что вне задачи.

### Хронология (по необходимости)

Краткие итерации до стабилизации решения, если без этого теряется смысл.

## Правила

1. Несколько решений одной задачи — **разделы одного файла**, не несколько файлов.
2. Номер файла — следующий свободный в `docs/adr/`; Issue — только в тексте.
3. Не дублируй Issue целиком — только интерпретации и выборы.
4. Без секретов, токенов, персональных данных.
5. Язык как у задачи (обычно русский); имена в коде — как в репозитории.
6. Не выдавай догадку за принятое решение — спроси Пользователя.
7. Содержание ADR должно соответствовать уже реализованному коду.

## Чего не делать

- Не подставлять номер Issue вместо `{NNN}` в имени файла.
- Не оставлять статус, который требует отдельного коммита Ведущего после merge.
- Не считать вердикт бота верификации MR заменой ADR.
- Не удалять старые ADR при замене решения.

## Итог `/adr`

Покажи путь к созданному/обновлённому файлу и кратко: номер ADR, статус, связанный Issue. Напомни: дальше можно оформлять MR (в чате — по инструкции `/task`, раздел «Далее»).
